<Section id="what-it-does" num={1} title="What this agent does and why">

The agent ingests a list of leads, researches each lead, drafts a short personalized cold email, and creates a Gmail draft only after a <GLink slug="dry-run">dry-run</GLink> review passes. It does not auto-send. The useful automation is upstream: research, synthesis, quality checks, idempotent draft creation, and an audit trail.

Generic outbound fails because the first sentence could apply to anyone. This agent forces specificity by making Potter research the input to every email and making the pre-flight gate reject obvious machine-written patterns. The human stays on the loop where judgment matters: approving the exact draft before Gmail receives it.

The shape that survives production:

- Lead in from CSV, webhook payload, or M23's funding monitor output.
- Potter research for person, company, and recent activity.
- Claude drafts a subject and a body in the user's voice.
- Pre-flight check blocks bad subject lines, banned phrases, extra sentence count, and em dashes.
- Gmail draft object is built locally with an <GLink slug="idempotency-key-per-lead">idempotency key per lead</GLink>.
- Dry-run lands in Slack.
- Human approval flips the write gate.
- Approved drafts are created in Gmail.

<Callout kind="info" title="Drafts, not sends.">
  Gmail draft creation is a safer boundary than automatic sending. The agent does the repetitive work; the human still controls whether and when the email leaves the mailbox.
</Callout>

</Section>

<Section id="user-story" num={2} title="The user story">

Monday morning. A list of 50 fresh leads from a webinar last week. The goal: 50 personalized draft emails in Gmail by Monday afternoon, each one referencing something specific from the lead's recent activity.

Without the agent: a day of manual research and writing. Open LinkedIn, read the profile, scan posts, search the company, draft a short email, paste into Gmail, repeat. Ten minutes per lead at best; longer when you actually research deeply.

With the agent: five minutes for the research and message generation across all 50 because Potter and Claude run in parallel. Thirty minutes for the human review: read each proposed subject and body, reject anything off-tone, approve the rest. The agent then creates Gmail drafts for approved leads.

The economics are defensible. Potter and Claude stay around cents per lead, and Gmail draft creation is free at normal volumes. The discipline is what makes it usable: voice rules, dry-run, idempotency, and audit logs.

</Section>

<Section id="tools" num={3} title="Tool layer: provider survey">

Three tool surfaces show up in the loop:

- **Potter MCP for research.** `potter_research_person` for role, tenure, and public profile context; `potter_research_company` for firmographic and recent company context; `potter_summarize_linkedin_posts` for recent post hooks; `potter_find_decision_maker` when the imported lead is not the right person.
- **Claude API for composition.** Sonnet writes the subject and body. The system prompt holds the voice rules and 3 to 5 few-shot examples; user content is the research blob and the lead. Output is plain text, maximum 3 body sentences.
- **Gmail API for drafts.** `users.drafts.create` writes a draft into the user's mailbox after approval. The agent builds the RFC 2822 message, encodes it, and stores an `X-Idempotency-Key` header so reruns can be reconciled.

<Section id="architecture" num={4} title="Architecture: dry-run, interactive, background">

The same loop runs in three modes:

- **`--dry-run`.** Mandatory by global rule. Logs everything that would happen without writing to Gmail. Slack receives a thread per batch with each proposed subject and body rendered as a review card.
- **Interactive.** Ad-hoc lists. You run the script with a CSV path; it processes the list; the dry-run lands in Slack; you approve.
- **Background.** Cron-driven or n8n-triggered batches, such as M23's Monday funding monitor flowing into Tuesday outreach. Same loop; different trigger.

Idempotency per lead uses the normalized email address as the key. A lead processed last week and present again this week is recognized. The agent can skip draft creation or update the review card instead of writing duplicates.

<Section id="loop" num={5} title="Building the loop">

A working sketch. The script reads a CSV; for each lead it researches, composes, validates, logs a dry-run, and creates a Gmail draft only when the write gate is open.

<CodeBlock lang="ts">{`import 'dotenv/config';
import { readFileSync } from 'fs';
import { parse } from 'csv-parse/sync';
import Anthropic from '@anthropic-ai/sdk';
import { Client as MCPClient } from '@modelcontextprotocol/sdk/client';
import { createGmailDraft } from './gmail';
import { postDraftReviewToSlack } from './slack';

const DRY_RUN = process.argv.includes('--dry-run');

const claude = new Anthropic();
const potter = await MCPClient.fromCommand('npx potter-mcp');

const SYSTEM = readFileSync('prompts/cold-email-system.md', 'utf-8');
const FEW_SHOTS = JSON.parse(readFileSync('prompts/few-shots.json', 'utf-8'));

async function processLead(row: {
  email: string;
  first_name: string;
  last_name: string;
  linkedin_url: string;
  company_domain: string;
}) {
  console.log({ event: 'lead.start', email: row.email });

  const [person, company, posts] = await Promise.all([
    potter.callTool({ name: 'potter_research_person', arguments: { linkedin_url: row.linkedin_url } }),
    potter.callTool({ name: 'potter_research_company', arguments: { domain: row.company_domain } }),
    potter.callTool({ name: 'potter_summarize_linkedin_posts', arguments: { linkedin_url: row.linkedin_url } })
  ]);

  const research = { person: person.content[0], company: company.content[0], posts: posts.content[0] };

  const response = await claude.messages.create({
    model: 'claude-sonnet-4-6',
    max_tokens: 300,
    system: [{ type: 'text', text: SYSTEM, cache_control: { type: 'ephemeral' } }],
    messages: [
      ...FEW_SHOTS,
      { role: 'user', content: 'Draft a cold email for this lead.\\n\\nResearch:\\n' + JSON.stringify(research) }
    ]
  });

  const text = (response.content.find((b) => b.type === 'text') as any).text.trim();
  const parsed = JSON.parse(text) as { subject: string; body: string };

  const check = preFlightCheck({ message: parsed.body, subject: parsed.subject });
  if (!check.ok) {
    console.warn({ event: 'lead.blocked', email: row.email, blockers: check.blockers });
    return;
  }

  const draft = buildEmailDraft({ lead: row, subject: parsed.subject, body: parsed.body });
  await postDraftReviewToSlack({ lead: row, draft, dryRun: true });

  if (DRY_RUN) {
    console.log({ event: 'lead.dry-run', email: row.email, subject: parsed.subject });
    return;
  }

  await createGmailDraft(draft);
  console.log({ event: 'lead.draft_created', email: row.email });
}

const rows = parse(readFileSync(process.argv[2], 'utf-8'), { columns: true });
for (const row of rows) await processLead(row);`}</CodeBlock>

<Section id="voice" num={6} title="Persona-matched cold email">

A model writing in your voice is a system-prompt problem, not a fine-tuning problem. The pattern that holds:

- **Rules in the system prompt.** Short sentences. No em dashes. No <GLink slug="llm-tell">LLM-tells</GLink> like "just wanted to", "i came across", or "i hope this". No markdown. Maximum 3 body sentences.
- **Few-shot examples.** Three to five real emails you have written and that worked. Pasted into the messages array as alternating user-content and assistant-content turns. The model learns tone faster from examples than from rules.
- **Hook reference required.** The system prompt requires the first sentence to reference something specific from the research blob: a recent post, product mention, hiring signal, funding event, or video.
- **Aggressive prompt caching.** The system prompt and few-shots are stable across the batch. Cache them; per-lead cost drops.

A sketched system prompt:

<CodeBlock lang="md">{`You write personalized cold emails in {USER}'s voice.

Voice rules:
- Short sentences. Direct. Specific.
- No em dashes anywhere.
- No "just wanted to", "i came across", "i hope this", "circling back", "reaching out to", or "touching base".
- No markdown formatting.
- Body is maximum 3 sentences total.
- Subject is non-empty and 80 characters or less.

Required:
- Reference one specific thing from the research blob.
- First body sentence is the personalization.
- Second body sentence is the value.
- Third body sentence is a soft CTA.
- Return JSON: { "subject": "...", "body": "..." }.`}</CodeBlock>

<Section id="gmail-drafts" num={7} title="Gmail draft creation">

Gmail draft creation is a clean final write because it separates drafting from sending. The agent creates the exact email in the user's mailbox, but the user can still edit or delete it before send.

The production recipe:

- **Build a plain object first.** `to`, `subject`, `body`, and `headers` are easier to validate before you encode the RFC 2822 message.
- **Idempotency header.** Add `X-Idempotency-Key` derived from the normalized email. Store that same key in your run log so reruns can detect duplicate drafts.
- **One Gmail write per approved lead.** Approval is the boundary. Draft creation does not run during dry-run.
- **Keep the body plain text.** Plain text survives inbox clients and review flows better than generated HTML.

<Section id="eval" num={8} title="Eval setup">

The golden set for this agent:

- **10 real past leads** for whom you actually wrote and sent cold emails. The original email you sent is the reference.
- **The same research inputs** that were available at the time. Snapshot Potter responses if they have changed since.
- **<a href="/m/evaluation#glossary-llm-as-judge">LLM-as-judge</a> rubric.** Score on tone, specificity, subject quality, length, and absence of LLM-tells. Each on a 1-5 scale; aggregate into a final score.
- **Hard fail on banned tokens.** Any body containing an em dash, "just wanted to", or "i came across" auto-fails regardless of judge score.

Run the eval before every meaningful change: new system prompt, new model version, new few-shots, or new research source. Watch the median score and the worst case; both have to hold.

<Section id="deploy" num={9} title="Deployment">

Two deployment shapes:

- **Standalone Node CLI.** The script above, run from a terminal. `node cold-email.js leads.csv --dry-run`. Easiest to use ad-hoc.
- **n8n workflow.** A trigger node feeds the lead list into a Code node that runs the same loop. Slack posts the dry-run thread. Approval comes through Slack interactivity routed back into the workflow.

The Gmail side needs OAuth with the `gmail.compose` scope for draft creation. Store refresh tokens in your secrets manager, not in the repo. Log the Gmail draft id after creation so the Slack review thread, run log, and mailbox can be reconciled.

</Section>

<Section id="what-broke" num={10} title="Iteration log: what broke">

The real cold-email failure modes from building this:

- **DKIM/SPF setup missing.** Emails landed in spam because the sending domain was not authenticated. Fix the domain setup before you measure copy quality.
- **Gmail rate limits at high volume.** Free accounts are capped at 500 sends/day. This module creates drafts, but the approved send path still needs volume-aware pacing.
- **Draft API quota.** Gmail draft writes consume API quota, and the relevant ceiling is 250 quota units per user per second. Batch draft creation with backoff instead of a firehose loop.
- **Apify rate limits during research phase.** The bottleneck can show up before Claude or Gmail. Queue Potter research, cache recent results, and retry with jitter.
- **Persona drift across the batch.** Without prompt caching, tone drifted as examples changed. Caching the system prompt and few-shot set kept the voice stable.
- **Few-shot bleed-through.** Specific phrases from examples appeared verbatim in new emails. Mitigation: rotate the few-shots periodically and reject repeated phrasing in evals.
- **Email normalization.** Case differences and whitespace created duplicate draft attempts. Normalize emails at ingest and use the normalized address for idempotency.

</Section>

<Section id="check" title="Check yourself">

<CheckQuestion id="m22-q1" question="1. Why is the dry-run flag mandatory before Gmail draft creation?">
Draft creation is an external write into a real mailbox. Dry-run logs to Slack for human review before that write happens. The cost of pausing is small; the cost of filling a mailbox with bad drafts is real.
</CheckQuestion>

<CheckQuestion id="m22-q2" question='2. What does "persona-matched" email generation mean and how do you achieve it?'>
Output that reads like the user wrote it. Achieved with a tight system prompt of voice rules plus 3 to 5 few-shot examples of the user's actual past emails. Prompt caching keeps the voice stable across the batch.
</CheckQuestion>

<CheckQuestion id="m22-q3" question="3. What subject-line rule does pre-flight enforce?">
Subject must be non-empty and 80 characters or less. Long or blank subjects are blocked before Gmail draft creation.
</CheckQuestion>

<CheckQuestion id="m22-q4" question="4. Why create Gmail drafts instead of sending automatically?">
Drafts keep the useful automation while preserving human control over final send. The agent handles research, writing, validation, and draft creation; the human can edit, delete, or send.
</CheckQuestion>

<CheckQuestion id="m22-q5" question="5. Cost per lead order of magnitude?">
About $0.10 in Potter providers plus about $0.04 in LLM with caching. Gmail API draft creation is free at normal volume. A 50-lead batch is roughly $7 in variable cost.
</CheckQuestion>

<CheckQuestion id="m22-q6" question="6. How do you keep voice consistent across a batch of 50?">
Cache the system prompt and few-shot examples. The stable prefix keeps voice rules and examples identical across leads while the per-lead research changes.
</CheckQuestion>

<CheckQuestion id="m22-q7" question="7. What banned phrases does the pre-flight check enforce?">
"just wanted to", "i came across", "i hope this", "circling back", "reaching out to", "touching base", plus em dashes and bodies over 3 sentences. The list grows when new tells appear in production output.
</CheckQuestion>

<CheckQuestion id="m22-q8" question="8. How does this agent compose with M23 and M24?">
M22's lead list can come from M23's funding monitor. M24's meeting prep can fire after the outbound thread books a meeting. The three form a loop: monitor surfaces companies, outreach drafts emails, meeting prep researches the room.
</CheckQuestion>
