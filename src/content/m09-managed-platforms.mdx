<Section id="what-managed" num={1} title='What "managed" actually means'>

[Module 07](/m/harnesses) was about handcrafted harnesses: Claude Code, Codex, Cursor, n8n, your own 50-line script. This module is the alternative path. A <GLink slug="managed-platform">managed agent platform</GLink> runs the loop *for you*. You hand it tools, prompts, and configuration; it returns sessions, evals, deployment, and observability.

What the platform takes care of, on average:

- **The loop itself.** No while-stop_reason loop in your code. The platform calls the API, dispatches tools, persists state.
- **Hosted state.** Sessions, threads, memory, conversation history, often persisted on the vendor's infrastructure.
- **Evals.** Built-in golden-test runners, regression suites, sometimes LLM-as-judge tooling.
- **Observability.** Dashboards for latency, cost, failure modes, success rates, eval scores. Plug in your own only if you outgrow theirs.
- **Deployment.** Hosted runtimes for the agent itself, often with auto-scaling and built-in rate limiting.
- **Multi-user and access control.** When the platform is the runtime, IAM is its problem, not yours.

What you can no longer change:

- **The loop's exact shape.** If the platform's loop does not stream the way you need, or batches differently, or has a hidden retry policy, you do not have a knob.
- **Hidden defaults.** Platforms apply prompt-shape munging, tool-output reformatting, retry policies, fallback models. Often invisible until something behaves unexpectedly.
- **Storage formats.** Your eval data, session logs, audit trails are in the platform's format. Migration cost is real.

The honest framing: managed platforms make standard shapes very fast and non-standard shapes painful. Pick deliberately.

</Section>

<Section id="claude-agent-sdk" num={2} title="Claude Agent SDK">

Anthropic's first-party SDK for building agents on top of the Claude API. Sits a layer above the basic `@anthropic-ai/sdk`: it gives you the agent class, tools API, hooks, memory primitives, and deployment helpers without forcing a hosted runtime.

What it bundles:

- **Agent class.** A typed wrapper around the loop. You define system prompt, tools, hooks; the agent runs.
- **Tools API.** Tool definitions are typed; tool implementations are typed; tool errors propagate as typed errors.
- **Hooks.** Lifecycle interceptors (similar to Claude Code hooks at a different layer): before-call, after-call, on-tool-use, on-error.
- **Memory primitives.** Hooks for short-term and long-term memory, with optional Anthropic-managed storage.
- **Deployment.** Run the agent on your own infrastructure; the SDK does not require a hosted runtime.

What it does not give you out of the box: a UI (you build that on top), a visual builder (Anthropic does not ship one as of 2026), full observability dashboards (integrate with your own).

When Claude Agent SDK wins:

- You are committed to Anthropic and want first-party Claude features (extended thinking, prompt caching, tool design at scale) integrated into the agent runtime.
- You want code-first agents in TypeScript or Python without wiring the loop yourself.
- You will run on your own infrastructure (VM, Lambda, Cloudflare Worker, etc).

When it loses: you want provider portability (Vercel AI SDK is closer to that), you want a visual builder (OpenAI Agent Builder or Agentspace), or you have enterprise GCP requirements (Agentspace).

</Section>

<Section id="openai-agent-builder" num={3} title="OpenAI Agent Builder">

OpenAI's hosted offering for building agents. Two surfaces: an SDK (similar shape to Claude Agent SDK) and a visual graph builder for composing agents node by node. The visual builder is the differentiator and the lock-in vector.

What it bundles:

- **SDK + visual builder.** Two surfaces, one underlying runtime.
- **Hosted threads, sessions, memory.** First-party state on OpenAI infrastructure.
- **Function calling integrated.** Tools you define are wired in to the agent loop.
- **Eval framework.** Golden tests, regression scoring, integrated into the platform UI.
- **Deployment on OpenAI infrastructure.** Limited self-hosting; export paths exist but are uneven across features.
- **Multi-agent orchestration.** Hand-off flows, sub-agents, role-based agents through the visual builder or the SDK.

When Agent Builder wins:

- OpenAI shop with a team that includes non-engineers who want to compose agents visually.
- Need built-in evals and deployment without standing up your own.
- Standard agent shapes (chat, research, customer support) where the visual builder's defaults match what you want.

When it loses: provider portability is a requirement (visual builder is OpenAI-only and hard to migrate), you have strict on-prem or VPC requirements, or your flow is non-standard enough that the visual builder fights you.

</Section>

<Section id="google-agentspace" num={4} title="Google Agentspace (now Gemini Enterprise Agent Platform)">

Google Cloud's enterprise agent platform built on Vertex AI. At Google Cloud Next 2026 the offering was rebranded into the broader Gemini Enterprise umbrella; the standalone "Agentspace" name is being phased out, but the capabilities described below carry over. Positioned for enterprise customers already on GCP, with strong integrations into Google Workspace (Gmail, Drive, Calendar) and enterprise identity (IAM, VPC).

What it bundles:

- **Vertex AI as the model substrate.** Multiple model providers hosted (Gemini, Anthropic via Vertex, third-party open-weights). Multi-provider on the model side.
- **GCP-native deployment.** Runs in your GCP project with VPC, IAM, audit logging, compliance tooling.
- **Workspace integrations.** Native connectors for Gmail, Drive, Calendar, Sheets, Docs.
- **Visual flow editor.** For composing multi-agent workflows.
- **Enterprise eval and observability.** Compliance-grade reporting; audit logs that integrate with GCP-wide monitoring.

When Agentspace wins:

- Enterprise team already on GCP with Workspace as the primary collaboration substrate.
- Compliance requirements (HIPAA, SOC 2, regional residency) that are easier on GCP than on a startup-grade platform.
- Cross-functional agents that need access to Workspace data through audited, IAM-controlled paths.

When it loses: you are not on GCP and not planning to migrate; team is small enough that enterprise overhead is a cost without payoff; your stack is Anthropic or OpenAI-aligned and Vertex's quirks are not worth absorbing.

</Section>

<Section id="vercel-ai-sdk" num={5} title="Vercel AI SDK">

Strictly speaking, Vercel AI SDK is not a managed agent platform. It is a provider-agnostic TypeScript SDK with React-friendly streaming primitives. It belongs in this comparison anyway because it covers similar ground (multi-provider, tool calls, streaming, hooks) without the hosted-state lock-in that the others carry.

What it bundles:

- **Provider abstraction.** Swap Anthropic, OpenAI, Google, open-weights providers with a one-line change. The lowest provider lock-in of any platform here.
- **Streaming-first React helpers.** Built for Next.js apps where the UI streams tokens as they arrive.
- **Tool calling support.** Tools defined in TypeScript, typed all the way through.
- **No hosted state.** You bring your own database; the SDK does not persist for you.
- **No first-party evals.** Integrate Promptfoo, custom test harness, or another eval framework.

When Vercel AI SDK wins:

- Building a Next.js or React app where streaming UI is core to the UX.
- Want provider portability and willing to maintain your own state, evals, and observability.
- Code-first culture; visual builders are not on the requirements list.

When it loses: you want a managed runtime (this is just a library), you need built-in evals, or you want an eval/observability dashboard out of the box.

</Section>

<Section id="lock-in" num={7} title="Lock-in dimensions">

A managed platform locks you in along several axes. Knowing which axes matter for you is what makes the build-vs-buy choice survivable.

The dimensions:

- **Provider lock-in.** Are you tied to one model provider? Claude Agent SDK is Anthropic-only; OpenAI Agent Builder is OpenAI-only; Vercel AI SDK is the most agnostic; Agentspace is multi-provider through Vertex but GCP-locked on infra.
- **Hosted-state lock-in.** Is your conversation state, memory, and audit trail stored on the vendor's servers in their format? OpenAI threads, Anthropic-managed sessions, Agentspace persistence: each one is its own export-and-port problem if you migrate.
- **Eval format lock-in.** Are your golden tests and eval reports in the platform's proprietary format? Some platforms (especially the visual-builder-driven ones) make eval data hard to extract.
- **Deployment-target lock-in.** Where does your agent run? Agentspace runs on GCP. OpenAI Agent Builder runs on OpenAI. Claude Agent SDK and Vercel AI SDK run anywhere.
- **API surface lock-in.** Is your application code calling platform-specific methods, or is it talking to your own abstraction? This is the one you control: even on a heavily managed platform, abstracting the agent loop behind your own interface keeps the migration cost bounded.

The general pattern: the more managed-ness you accept, the more lock-in you sign up for. Built-in evals and visual builders are the two features that quietly add the most lock-in.

</Section>

<Section id="build-vs-buy" num={8} title="Build vs buy heuristics">

The cleanest signals for "buy":

- Need built-in evals and deployment fast.
- Limited engineering bandwidth (solo, small team, junior team without prior agent experience).
- Standard flow shape (chat agent, research, customer support, knowledge-base Q&A).
- Time-to-market matters more than future migration cost.
- Non-engineers on the team will be composing flows.

The cleanest signals for "build" (raw SDK or custom harness):

- Custom flow shape that does not match any platform's defaults.
- Strict latency budgets where every layer needs to be a target for tuning.
- On-prem or strict data-residency requirements that managed platforms cannot meet.
- High volume where managed platform pricing becomes punitive.
- Embedded in another product where the agent is a feature, not a tool.

<Section id="migrating" num={9} title="Migrating off a managed platform">

Migration cost is real but bounded. The practical strategy:

1. **Catalog what the platform owned.** Hosted state, evals, observability, deployment, IAM. Each is a separate replacement project.
2. **Identify the seam.** The interface your application code talked to. If you abstracted "the agent" behind your own interface from day one, the seam is clean. If you peppered platform-specific calls everywhere, the seam is the codebase.
3. **Build the replacement.** For each owned primitive, pick the swap. Hosted state to your DB; built-in evals to Promptfoo; observability to Datadog; deployment to your CI/CD.
4. **Run dual-stack.** Send a fraction of traffic through the new path while the platform serves production. Diff outputs. Catch hidden behaviors.
5. **Cut over and decommission.** Switch traffic; keep the platform around as a fallback for two to four weeks; decommission once confident.

<MigrationStepper />

The hidden tax: replacing the platform's UI surfaces (eval reporters for stakeholders, customer-data integrations, role-based access UIs) takes more eng-weeks than replacing the loop did. Realistic budget for a small-to-mid system: 2 to 6 months of focused work. Plan accordingly; do not promise a one-month migration when scope analysis would say six.

The defensive strategy from day one: abstract the agent loop behind your own interface. Treat the platform as an implementation detail, not the API your code knows about. This caps the migration cost at "swap one implementation"; it does not eliminate the work, but it bounds it.

</Section>

<Section id="check" title="Check yourself">

<CheckQuestion id="m09-q1" question="1. What is the cleanest signal that you should pick a managed platform?">
Need built-in evals and deployment fast, with limited engineering bandwidth, and your flow fits a standard shape (chat, research, customer support). When the alternative is "we will rebuild eval, deployment, and observability badly in three months," buying is right. The platform's UX surface for non-engineers is often the deciding factor in small teams.
</CheckQuestion>

<CheckQuestion id="m09-q2" question="2. What is the cleanest signal that managed platforms will hurt you?">
Custom flow shape that does not match defaults, multi-step orchestration with non-standard sequencing, latency budgets that need raw control over every layer, strict on-prem or data-residency requirements, very high volume where managed pricing becomes punitive, or embedding the agent inside a product where it is a feature rather than a standalone tool.
</CheckQuestion>

<CheckQuestion id="m09-q3" question="3. How does provider lock-in differ between Claude Agent SDK and Vercel AI SDK?">
Claude Agent SDK is Anthropic-only by design. Switching providers means rewriting on a different SDK. Vercel AI SDK is provider-agnostic: you can swap Anthropic, OpenAI, Google, or open-weights with a one-line provider change. Pick Claude Agent SDK when you want first-party Claude features integrated; pick Vercel AI SDK when provider portability is the priority.
</CheckQuestion>

<CheckQuestion id="m09-q4" question="4. Why do most managed platforms ship visual builders?">
Two reasons. They lower the bar for non-engineers to compose agents, which broadens the target market. And they hide the platform's underlying API surface, which raises switching cost: visual-builder artifacts do not port to other platforms. The first reason is for the customer; the second is for the vendor.
</CheckQuestion>

<CheckQuestion id="m09-q5" question="5. A startup wants to ship in two weeks with strong evals and zero ops. Recommend?">
Pick a managed platform: Claude Agent SDK if Anthropic-aligned and you can run on your own infra, OpenAI Agent Builder if you want hosted runtime plus visual builder, Vercel AI SDK if you are building a Next.js app with React streaming. Do not build the loop yourself. The two-week ship is real; the rebuild three months later is the avoidable failure.
</CheckQuestion>

<CheckQuestion id="m09-q6" question="6. An enterprise client requires data residency in their VPC. Recommend?">
Raw SDK (Module 06) on their infrastructure, or a managed platform that explicitly supports BYOC: Vertex AI behind Agentspace runs in your GCP project; AWS Bedrock can host Claude in your AWS account; Azure OpenAI runs in Azure tenants. Confirm the residency story before signing anything; "supports BYOC" varies in scope across vendors.
</CheckQuestion>

<CheckQuestion id="m09-q7" question='7. Is "no managed platform = always better" a defensible position?'>
No. Managed platforms win when your flow is standard and the alternative is reinventing eval, deployment, and observability poorly. Build-vs-buy is a context decision: solo dev with a custom flow on a tight latency budget should build; small team with a standard chat flow should buy. The mistake is treating either as the universal answer.
</CheckQuestion>

<CheckQuestion id="m09-q8" question="8. How do you keep migration off a managed platform feasible later?">
Abstract the agent loop behind your own interface from day one. Treat the platform as an implementation detail; your application code should talk to your interface, not directly to platform-specific methods. This caps migration cost at "swap one implementation." It does not eliminate the cost (you still rebuild evals, observability, deployment), but it bounds it.
</CheckQuestion>
