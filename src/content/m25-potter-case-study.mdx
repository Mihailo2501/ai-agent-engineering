<Section id="what-it-is" num={1} title="What Potter is and is not">

Potter is an MCP server. A tool layer. Roughly 25 tools that fetch data: company research, person research, LinkedIn enrichment, web scrape, web search, structured browser actions. Pure data fetching, schema-typed inputs and outputs, structured errors the agent can recover from.

<Callout kind="info" title="The repo">
Potter is open source under MIT. Source, capability spec, tool catalog, and the three runtimes all live at <a href="https://github.com/Mihailo2501/potter-mcp" target="_blank" rel="noreferrer noopener" className="text-accent-coral underline">github.com/Mihailo2501/potter-mcp</a>. Read alongside this module if you want to see the patterns described here in actual code.
</Callout>

What Potter is not:

- **Not a managed agent platform.** No agent loop runs inside Potter. The harness above Potter (Claude Code, n8n, a Node script) owns the loop.
- **Not a harness.** Potter does not consume models. The harness picks the model; Potter handles the data the model asks for.
- **Not opinionated about model choice.** Claude, GPT, Gemini, Llama via Ollama: the MCP protocol does not care.
- **Not stateful.** Each tool call is independent. Memory belongs to the harness.

The cleanest framing: Potter is a callable. The harness decides when and how to call it. This is what makes it portable across the three GTM agents in M22 to M24, and across any MCP-speaking environment a user might pick.

</Section>

<Section id="spec-discipline" num={2} title="The capability spec discipline">

Building Potter forced answers to most of the questions earlier modules raised. Where does the loop run? In the harness. Where do tools live? In an MCP server. What about state? In the harness. Why three runtimes? Different deployment stories.

The artifact that made the project tractable was the <GLink slug="capability-spec">capability spec</GLink>. One document per major version. The shape:

- **Goal.** What this version of Potter must do, in one sentence.
- **In scope.** Tool families and specific tools. Provider integrations. Runtime shapes.
- **Out of scope.** Things explicitly not in this version. Often longer than the in-scope list.
- **Acceptance criteria.** Concrete, testable bullets. The version is done when the criteria pass.

Why this matters:

- **Coding-agent compatibility.** Module 21 covered spec-driven coding. Potter is the lived example. Each spec was handed to Claude Code; the agent implemented against it; tests caught the cases where the implementation did not match the spec.
- **Scope control.** Without the spec, the project drifted. With it, every "while we're at it" suggestion got tested against the in-scope list. Most got rejected; a few got promoted to the next version.
- **Documentation by accident.** The spec is the documentation. New consumers read it to understand what Potter does; new contributors read it to understand what Potter does not do.

This is the discipline Module 21 recommends for any non-trivial coding work. Potter is the case study for what it looks like applied across 18 months of incremental builds.

</Section>

<Section id="three-runtimes" num={3} title="Three runtimes on one npm package">

The same core (the tool functions) ships under three different runtime wrappers:

<ThreeRuntimeDemo />

Why each exists:

- **MCP server.** The primary delivery shape. Used inside Claude Code, Cursor, n8n, or any MCP-speaking harness. The harness owns the loop; Potter ships the tool surface. This is how the three GTM agents in M22 to M24 consume Potter.
- **batch-brief runtime.** A scheduled host wrapped around the tool functions. Reads a CSV or queue, calls Potter tools per row, produces a markdown brief, posts to Slack or saves to disk. Cron-driven or Inngest-triggered. M15's lessons in concrete form.
- **Standalone CLI.** Direct surface over the tool functions for one-off interactive use. `potter research-company --domain acme.com`. Useful for debugging and for ad-hoc spot checks without firing up a harness.

The shared core is what makes this tractable. The tool functions are written once. Each runtime wraps them differently: MCP exposes them via the protocol; batch-brief schedules invocations; the CLI parses argv and calls them directly. Adding a new tool means writing one function; all three runtimes pick it up.

<Section id="catalog" num={4} title="The 25-tool catalog">

Tools are namespaced by family:

- **`potter_research_*`.** Per-entity research: company, person, decision-maker discovery.
- **`potter_browser_*`.** Browser actions: open, click, fill, scroll, screenshot, extract, plus the special-case act.
- **`potter_linkedin_*`.** LinkedIn-specific: profile, company page, posts, post summarization.
- **`potter_web_*`.** Generic web: search, scrape, crawl.
- **`potter_extract_structured`** for ad-hoc structured extraction.
- **`potter_find_decision_maker`** for org-chart traversal.

Naming follows Module 10's discipline:

- **Namespace prefix.** `potter_*` so the model groups them mentally and the harness's tool dispatcher routes correctly.
- **Verb-then-noun.** `research_company`, `find_decision_maker`, `summarize_linkedin_posts`. Reads as actions.
- **Snake case.** Consistent across the catalog.

Each tool description carries:

- **What it does.** One sentence.
- **When to call it.** The shape of inputs that fit. "Use when the user asks about funding history, latest raises, or investor lists."
- **When NOT to call it.** Disambiguation against neighbors. "Do not use for general firmographic data; use `potter_research_company` for that."
- **Output shape hint.** What the agent gets back, at the level the agent needs to know.

<Section id="browser-act" num={5} title="The browser-act exception">

`potter_browser_act` is the one place where Potter calls an LLM internally. Module 13 covered the principle; this section covers the implementation.

The tool's job: navigate dynamic UIs (Cloudflare challenges, login redirects, A/B-rotating layouts) where deterministic selectors fail.

The flow:

1. Tool receives a goal: "click the Sign In button" or "find the price on this product page".
2. The tool's runtime takes a screenshot of the current page state via Stagehand.
3. The runtime calls Claude with a tightly-scoped prompt: "Given this screenshot and goal, choose the next action."
4. Claude returns an action (click + selector, fill + value, scroll, navigate).
5. Stagehand executes the action.
6. The tool returns a structured result to the harness.

The harness's main loop never sees the screenshot. The model decision is scoped, isolated, and disposable. The agent gets a clean structured result back.

Why this is the only exception:

- **Screenshots are expensive.** 1,500 to 3,000 tokens each. Returning them to the harness's loop on every browser action would inflate context dramatically.
- **Vision capability is not portable.** Some harnesses are not vision-capable. Coupling Potter's browser tool to the harness's vision means Potter only works in some harnesses.
- **The decision is small.** The model is choosing a Stagehand primitive, not making a multi-step plan. Scoped LLM call inside the tool is the right grain.

The discipline that makes this safe:

- **Tightly scoped prompt.** The model is told exactly what action shapes are valid; it is not free to reason about anything else.
- **Bounded action set.** Click, fill, scroll, navigate. Not arbitrary code execution.
- **Logged and budgeted.** Each browser-act call is metered; cost caps apply; structured events go to the audit trail.

</Section>

<Section id="routing" num={6} title="Provider routing and fallbacks">

Potter's tools route to providers. The default routing:

- **Proxycurl** for LinkedIn data (research_person, linkedin_profile, linkedin_company, linkedin_posts).
- **Apify** as fallback for LinkedIn data when Proxycurl 429s, plus primary for some scraping flows.
- **Firecrawl** for general web scrape and crawl.
- **Stagehand + Browserbase** for browser actions.
- **Custom search providers** for web_search.

The fallback shape: each tool declares a primary provider plus an ordered list of secondaries. On primary failure (rate limit, 5xx, timeout), the tool routes to the next available provider, normalizes the result back to Potter's output shape (so the consumer cannot tell which provider served the call), and returns.

The result: tools degrade gracefully. A Proxycurl outage does not stop research; it routes through Apify. The harness sees the same response shape; the audit trail records which provider served the call so observability can detect rising fallback rates.

</Section>

<Section id="what-broke" num={8} title="What broke during the build, what got rewritten">

Real lessons from the lived build:

- **Tool descriptions started too short.** Early descriptions read like one-liners. The model picked them inconsistently. The fix was the discipline this course teaches: descriptions are prompts; longer with explicit when/do-not beats works better.
- **Output shapes drifted across providers.** Proxycurl returned slightly different fields than Apify; the agent had to handle two shapes. The fix was provider-side normalization at the Potter boundary so the agent always sees one shape.
- **Errors were free text.** Early `tool_result.is_error` events returned "request failed: 500". The agent did not know whether to retry, fall back, or abort. Structured errors with kind ([Module 10](/m/tool-design)) were retrofitted; recovery improved.
- **Memory crept into tools.** A first attempt at "remember the last query" inside a tool broke the no-state rule. Removed; memory moved to the harness; portability returned.
- **The browser-act exception almost grew.** Several tools were proposed with internal LLM calls "for similar reasons". Each was rejected after weighing the alternative; only browser-act met the bar. The discipline of "what is the alternative" kept the exception singular.
- **batch-brief originally lived as a separate package.** Splitting it into its own npm package made shared utility code awkward. Merged back into the main repo as a separate runtime entry; cleaner.

</Section>

<Section id="deployed" num={9} title="The three runtimes deployed">

How each runtime actually ships:

- **MCP server.** Local stdio inside Claude Code (`.mcp.json` references `npx potter-mcp`). HTTP transport for remote use (`https://mcp.potter.example/mcp`). Auth varies by deployment.
- **batch-brief runtime.** Inngest workflow consuming a Postgres queue or scheduled cron. Reads a CSV or DB table, calls Potter tools per row in parallel, persists results, posts a Slack digest plus PDF artifacts to R2. M15 patterns concretely.
- **Standalone CLI.** Distributed via npm; runs anywhere there is Node. `npx potter research-company --domain acme.com` for ad-hoc use; scripted from shell when needed.

Each runtime emits structured events to the same logging surface. The audit trail does not care which runtime served the call; the events look the same.

</Section>

<Section id="what-you-do" num={10} title="What you do with this">

The course's lessons assemble into Potter. They also assemble into M22, M23, and M24. The reader's homework is to assemble them into something of their own.

A few patterns worth carrying:

- **Write the spec first.** [M21](/m/coding-agents) is the rule; Potter is the proof.
- **Tools are descriptions plus schemas.** [M10](/m/tool-design). Potter's catalog is what tool design at scale looks like in production.
- **Memory belongs to the harness.** [M11](/m/memory-retrieval). Potter has none; that is what makes it portable.
- **Structured outputs everywhere.** [M12](/m/structured-output). Every Potter call returns typed data; every consumer parses without surprises.
- **Background work is its own runtime.** [M15](/m/async-agents). batch-brief is what M15 looks like wrapped around a tool layer.
- **Cost lives in the harness; tool layers are cheap.** [M17](/m/cost-economics). Potter's cost is provider fees; the harness pays the LLM bill.
- **Narrow surfaces are easier to defend.** [M18](/m/security-adversarial). Potter does not write to anything outside its declared providers.
- **Observability is shipped events; aggregation lives at the edge.** [M19](/m/observability). Potter logs; the harness aggregates.

You do not need to build Potter. You can build something narrower or broader; you can build it on top of different providers; you can pick a different harness. The course's lessons survive all of that. The point is the discipline, not this specific tool.

</Section>

<Section id="check" title="Check yourself">

<CheckQuestion id="m25-q1" question="1. Why three runtimes, not one?">
Different consumption shapes. MCP for inside-harness use, batch-brief for scheduled jobs, CLI for direct interactive use. One core (the tool functions) under three wrappers; each wrapper handles its own input shape and delivery. Adding a tool means adding a function; all three runtimes pick it up.
</CheckQuestion>

<CheckQuestion id="m25-q2" question="2. Why is the capability spec the load-bearing artifact?">
It locks acceptance criteria before implementation. The coding agent (Claude Code) implements against it; reviews check the diff against the spec; future contributors read the spec to learn what is in scope. Without it, the project drifts and every "while we're at it" suggestion becomes a fight.
</CheckQuestion>

<CheckQuestion id="m25-q3" question='3. What does "no LLM inside MCP except browser-act" actually buy you?'>
Portability. Potter runs identically in any MCP-speaking harness because the harness owns the loop. Memory and model choice live one layer up. Coupling LLM calls into the tool layer would make Potter expensive, slow, and tied to specific harness vision capabilities; the rule prevents it.
</CheckQuestion>

<CheckQuestion id="m25-q4" question="4. Why is browser-act the exception?">
Returning screenshots through MCP and back to the harness's loop inflates context (1,500 to 3,000 tokens each) and couples Potter to the harness's vision capabilities. A scoped LLM call inside the tool is cheaper and more reliable for that specific case. The rule survives because the alternative is genuinely worse.
</CheckQuestion>

<CheckQuestion id="m25-q5" question="5. How does Potter handle provider failures?">
Routing with fallback. Each tool declares a primary provider plus an ordered list of secondaries. On primary failure (rate limit, 5xx, timeout), the tool routes to the next available provider, normalizes the result back to Potter's output shape so the consumer cannot tell which provider served the call, and returns. The audit trail records which provider served each call so observability can detect rising fallback rates.
</CheckQuestion>

<CheckQuestion id="m25-q6" question="6. What lives in the harness, not in Potter, and why?">
Memory, model choice, audit logging, user authentication, rate limiting, scheduling. Harness-layer concerns. Coupling them to Potter would break portability across harnesses. Potter ships clean structured tool calls; the harness owns the rest of the experience.
</CheckQuestion>

<CheckQuestion id="m25-q7" question="7. A reader wants to build something like Potter. First step?">
Write the capability spec. Lock acceptance criteria. Start with 3 to 5 tools maximum; grow only when the spec demands it. The discipline of the spec is what makes the project tractable to build with a coding agent and tractable to maintain after.
</CheckQuestion>

<CheckQuestion id="m25-q8" question="8. How does Potter relate to M22-M24?">
M22 (LinkedIn outbound) calls Potter for per-lead research. M23 (funding monitor) calls Potter for per-company deep research. M24 (meeting prep) calls Potter for per-attendee briefs. Three different agents, one Potter. The same tool surface powers three workflows that look unrelated at first glance.
</CheckQuestion>
