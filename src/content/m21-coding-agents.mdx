<Section id="what-makes-different" num={1} title="What makes a coding agent different">

A general agent decides what tool to call. A <GLink slug="coding-agent">coding agent</GLink> decides what change to make. The shape of the work is genuinely different:

- **File system as state.** The agent's "memory" is the working tree. Edits persist across turns; a bad edit is visible immediately; the next turn reads the modified files.
- **Tests as eval.** The cheapest, most reliable feedback signal is "the test suite passes". Iterating against tests is much faster than iterating against open-ended judgments.
- **Git as reversibility.** Every change is recoverable. The agent can experiment more freely than agents that touch external systems with non-reversible side effects.
- **Iteration cost.** Running tests is the slow loop. CI pipelines, linting, type-checking are all part of the wall time. The agent's tool choices have to account for this.

These properties make coding agents tractable in ways general agents are not. They also make benchmarks (SWE-bench specifically) more meaningful: a coding agent's success or failure can be measured objectively against a real-world task, in a way that "did the agent help with the meeting" cannot.

</Section>

<Section id="claude-code-depth" num={2} title="Claude Code in depth">

[Module 8](/m/claude-code-surface) covered the extension surface (skills, plugins, hooks, slash commands). This section is about the workflow patterns.

The patterns that actually ship features:

- **Spec-then-implement.** Write the spec in a doc, hand it to Claude Code, ask for a plan, review the plan, then implement. The spec discipline is what makes the agent's output predictable.
- **Test-driven.** Tell the agent to write tests first. The tests become the acceptance criteria; the implementation has to satisfy them. This catches misunderstandings before they become buggy code.
- **Plan-mode-then-edit.** Use plan mode (the model writes the plan but does not edit) for non-trivial work. Review the plan; ask follow-up questions; then exit plan mode and let the agent execute.
- **Subagents for boundary work.** Spin up a subagent for a research pass (read the codebase, find all the call sites, summarize) before the main agent makes edits. Keeps the main agent's context focused.
- **Hooks for guardrails.** PostToolUse hooks running tests on every edit, PreToolUse hooks blocking accidental destructive commands. The harness enforces what the agent should not be trusted to enforce alone.

A practical example, in a single session:

<CodeBlock lang="md">{`User: Read specs/sign-in-form.md and write a plan to implement it.
Claude Code: [reads spec, writes plan, lists files to touch]

User: Plan looks good. Implement it. Make sure to add the tests first.
Claude Code: [creates test files, then implementation, runs tests, fixes failures]

User: Run the linter.
Claude Code: [runs linter, fixes warnings]

User: Open a PR.
Claude Code: [opens PR with summary linking to spec]`}</CodeBlock>

The discipline: short turns, explicit acceptance criteria, the spec as the contract.

</Section>

<Section id="codex" num={3} title="Codex CLI">

OpenAI's <GLink slug="codex-cli">Codex CLI</GLink>. The two everyday entry points:

- **Interactive mode.** Run `codex` in a terminal for a chat-and-edit loop similar to Claude Code's interactive session.
- **`codex exec` (non-interactive).** Pass a prompt; the agent runs autonomously, edits files, runs tests, and summarizes. This is the entry point used for CI integrations and for review-shaped runs (e.g., `codex exec "review the diff against main and flag issues"`). The Claude Code `codex-plugin-cc` plugin wraps this exec path so you can launch Codex review runs from inside a Claude Code session.

Where Codex specifically wins:

- **Image-to-code.** OpenAI's vision-strong models pair with Codex for "here's a screenshot of the design; build it." Useful for UI scaffolding from mockups.
- **Broad code reviews.** Cross-file consistency checks, design-doc audits, the "second opinion" pass that catches things you missed.
- **Plugin ecosystem.** Codex has its own plugin model (similar in spirit to Claude Code's, distinct in details). Useful when you need OpenAI-specific integrations.

When Claude Code beats Codex:

- **Tool-use reliability for agent loops.** Claude Code's stop-and-tool patterns are mature.
- **Anthropic-aligned stacks.** When your model preference is Claude, the harness alignment matters.
- **Extension surface depth.** Hooks, skills, plugins, MCP attachment all in one place ([Module 8](/m/claude-code-surface)).

</Section>

<Section id="cursor" num={4} title="Cursor">

<GLink slug="cursor-agent">Cursor</GLink> is an editor (a VS Code fork) with a built-in agent mode. The shape that distinguishes it:

- **In-editor pair-programming.** The agent works alongside you in real-time. Inline suggestions, agent runs that edit files in place, chat that has full repo context.
- **Rules files.** Per-project agent configuration lives in `.cursor/rules/*.mdc` (the current format; the older `.cursorrules` single-file form is deprecated and being phased out). Each `.mdc` file holds coding style, libraries to prefer, things to avoid. Maintains agent behavior across sessions and team members.
- **Agent vs chat.** Cursor distinguishes "chat" (you ask, it answers) from "agent mode" (it runs autonomously, edits files, runs commands). Both have their place.

When Cursor wins:

- **Real-time iteration.** You want feedback as you type, agent runs that touch the files you are looking at, chat with full repo context without leaving the editor.
- **Team-shared rules.** A repo with established patterns benefits from `.cursor/rules/*.mdc` files that every team member's Cursor reads.
- **GUI-driven workflows.** Engineers who prefer an editor over a terminal find Cursor a more natural fit than Claude Code or Aider.

When Cursor loses:

- **Heavy automation workflows.** When you want hooks, scheduled tasks, or programmatic agent runs, terminal-native tools (Claude Code, Aider) fit better.
- **Lightweight repos.** The editor overhead is not always worth it for quick CLI-shaped work.

</Section>

<Section id="aider" num={5} title="Aider">

<GLink slug="aider">Aider</GLink> is a terminal-driven coding agent with a strict diff-based edit workflow.

What sets it apart:

- **<GLink slug="diff-style">Diff-style edits</GLink>.** The agent's output is expressed as diffs to the working tree. Applied atomically; rejected if the patch does not apply cleanly. Forces the agent to be specific about what it is changing.
- **REPL-shaped UX.** You stay in a chat loop in the terminal; ask for changes; review; continue. Lower friction than Claude Code's session model for one-off edits.
- **Repo-aware out of the box.** Sends a repo map to the model so it knows the project structure without you having to tell it.
- **Provider-flexible.** Works with Claude, GPT, Gemini, OpenRouter, local models. Useful when you want a coding agent without committing to one provider.

When Aider wins:

- **Quick edits.** Open the terminal, describe the change, accept the diff. Lower overhead than starting a Claude Code or Cursor session.
- **Provider portability.** Switch models with a config flag.
- **Minimal-surface workflows.** Engineers who want a chat-and-diff loop without the additional surface area of a full IDE or extension framework.

When Aider loses:

- **Long-form planning.** Other tools have richer plan modes and subagent patterns.
- **Hook-driven CI integration.** Claude Code's hook surface is more mature.

</Section>

<Section id="devin" num={6} title="Devin and the autonomous-engineer lineage">

<GLink slug="devin">Devin</GLink> launched the "autonomous engineer" category: a long-running agent that takes a task description, works on it for hours, and opens its own pull request. Cognition's framing was "AI software engineer"; the SWE-bench numbers got attention; the actual product behavior got more nuanced reviews.

The shape:

- **Hosted.** You log into a UI, file a task, watch the agent work. Not a CLI you run locally.
- **Long-running.** Hours per task. The agent reads the codebase, plans, edits, tests, iterates.
- **<GLink slug="autonomous-pr">Autonomous PR</GLink> at the end.** The deliverable is a GitHub PR with the agent's diff and summary. You review.

Where Devin-class agents work well:

- **Well-specified medium-sized tasks.** Add a feature with clear requirements; refactor a bounded piece; write tests for a module.
- **Background work queues.** You file a batch of tasks; the agent works through them; you review the PRs.
- **First-pass implementation.** Even when the result needs revision, getting to "first draft of the change" is often the hardest part.

Where they consistently fail:

- **Implicit specs.** "Make this better" is not enough; the agent invents requirements that may not match yours.
- **Domain-knowledge-heavy tasks.** When the right answer requires understanding business context the codebase does not explain, the agent flounders.
- **Complex debugging.** The bug-fixing loop on production-grade systems often requires the kind of reasoning that the agent does not yet do reliably.

The 2026 lineage includes several entrants in this space; capabilities and quality are improving. The right framing today: useful for queueing work, not a replacement for engineering judgment.

</Section>

<Section id="swe-bench" num={7} title="SWE-bench and the eval shape">

<GLink slug="swe-bench">SWE-bench</GLink> is a public benchmark that takes real GitHub issues from open-source projects, has the agent attempt to fix them, and scores by whether the resulting patch makes the project's tests pass. SWE-bench Verified is a curated subset cleaned of ambiguous cases.

Why it matters:

- **Real-world tasks.** Not synthetic; not toy. Issues from real projects with real tests.
- **Objective scoring.** A test suite passes or it does not. No LLM-judge bias.
- **Public scoreboard.** Comparison across coding agent products and approaches.

The caveats:

- **One benchmark.** Real codebases have shapes SWE-bench does not test. Frontend-heavy work, long-context cross-cutting changes, debugging-heavy workflows are underrepresented.
- **Test quality varies.** Some passing patches are technically correct but architecturally bad. The benchmark cannot judge that.
- **Context window matters.** Higher SWE-bench scores often come with larger context budgets, which production agents may not have.
- **Test gaming.** Some agents have learned to detect SWE-bench-style tasks and behave specifically for them. Read scores critically.

The honest framing: SWE-bench scores are useful as a directional signal for a category of work. Stating "this agent scores X%" tells you something; it does not tell you "this agent will improve your team by Y%". Verify on your own codebase with your own evals before committing.

</Section>

<Section id="spec-driven" num={8} title="Spec-driven coding workflows">

<GLink slug="spec-driven-coding">Spec-driven coding</GLink> is the workflow that makes coding agents reliable. The core move: write the spec first, then have the agent implement against it.

Why it works:

- **Reduces "looks plausible but wrong" rate.** A spec tells the agent (and you) what success means before any code is written. Without it, the agent invents requirements; with it, you catch misunderstandings cheaply.
- **Makes review cheaper.** Reading a diff against a known spec is fast; reading a diff to figure out what it was supposed to do is slow.
- **Becomes the eval.** The spec's acceptance criteria become behavioral test cases ([Module 16](/m/evaluation)); they catch regressions in CI.
- **Surfaces ambiguity.** Writing the spec forces you to decide things you might otherwise punt on. The act of writing reveals where the requirements are unclear.

The minimum viable spec:

<CodeBlock lang="md">{`# Goal
One sentence: what does success look like?

# In scope
- specific feature 1
- specific feature 2

# Out of scope
- thing the agent should NOT do
- thing that is part of a different ticket

# Acceptance criteria
- [ ] testable bullet 1
- [ ] testable bullet 2
- [ ] testable bullet 3
- [ ] (at least 3, more for non-trivial work)`}</CodeBlock>

<Section id="multi-agent" num={9} title="Multi-agent for coding">

The planner / implementer / reviewer pattern: one agent plans, one implements, one reviews. Sometimes useful; often overhead.

When it pays off:

- **Larger PRs.** When the work is too big for one context, splitting across agents (each with its own context budget) lets the system handle bigger tasks.
- **Independent perspectives.** A reviewer agent that did not see the implementation work catches different things than the implementer would catch about its own work.
- **Specialization.** A planner trained on planning, an implementer trained on implementation, a reviewer trained on review. The current state is "same model, different prompts" but the framing is useful even there.

When it is overhead:

- **Smaller changes.** A 50-line bug fix does not need three agents.
- **Tight feedback loops.** When you want iteration speed, multi-agent introduces handoff latency.
- **Shared context costs.** Running three agents on the same codebase means three reads of the codebase, three context budgets, three sets of API costs.

The pattern that often works: one agent in plan mode produces the plan, you review it, then a single agent implements. The "second pass" review is cheaper as a separate agent run after the implementation, when you already have the diff to look at.

<Section id="picking" num={10} title="Picking a coding agent for the job">

The decision tree, condensed:

- **Solo dev iterating in a terminal.** Claude Code or Aider. Claude Code if you want the extension surface (hooks, skills, plugins); Aider if you want minimal-surface diff-based edits.
- **Pair-programming in an IDE.** Cursor.
- **Broad code review or image-to-code.** Codex.
- **Long-running autonomous PR.** Devin or one of the autonomous-PR successors.
- **Mixed.** Most working engineers use multiple. Claude Code for daily implementation work; Cursor for in-editor pair sessions; Codex for review passes; Devin for queued background tasks.

<CodingAgentMatrixDemo />

The honest framing: the tool that fits is the tool you actually open every day. A team that aligns on Cursor will out-ship a team that picks the "objectively best" tool that no one uses. Pick by daily-use comfort first; capability second.

</Section>

<Section id="check" title="Check yourself">

<CheckQuestion id="m21-q1" question="1. What makes a coding agent different from a general agent?">
File system as state, tests as eval, git as reversibility, iteration cost (running tests is the slow loop). The shape of "the work" is genuinely different from a general agent's tool decisions, which makes coding agents tractable in ways general agents are not.
</CheckQuestion>

<CheckQuestion id="m21-q2" question="2. When does Aider beat Claude Code?">
When you want minimal-friction quick edits via diff, especially in environments without the full Claude Code extension surface. Aider's REPL-shaped UX has lower overhead for one-off edits. Claude Code wins when you want hooks, skills, plugins, or MCP attachment.
</CheckQuestion>

<CheckQuestion id="m21-q3" question="3. SWE-bench score caveat?">
It is one benchmark, on real but specific GitHub issues. Scores may not transfer to your codebase, your stack, or your kind of work. Read scores critically: a high SWE-bench number is a directional signal, not a guarantee the agent will improve your team.
</CheckQuestion>

<CheckQuestion id="m21-q4" question="4. Spec-driven coding: why bother?">
Tells the agent (and you) what success means before any code is written. Reduces the "looks plausible but wrong" rate. Makes review faster because you can compare the diff against the spec instead of figuring out what the change was for. The spec also becomes the basis for behavioral tests.
</CheckQuestion>

<CheckQuestion id="m21-q5" question="5. Multi-agent for coding: when does it pay off?">
Larger PRs where planner / implementer / reviewer benefit from separated contexts. Independent reviewer perspectives that catch what the implementer missed. Smaller changes: overhead beats value; one agent in plan mode plus one implementation pass is usually faster.
</CheckQuestion>

<CheckQuestion id="m21-q6" question="6. Why is Potter's case relevant to M21?">
Built using Claude Code as the primary harness. Spec discipline (one capability spec per major version, concrete acceptance criteria) is what made a 25-tool MCP server tractable as agent-built work. The pattern of spec-first-implement-second is what M21 recommends, and Potter is the case study for it.
</CheckQuestion>

<CheckQuestion id="m21-q7" question="7. Pair-program agent vs autonomous PR: when each?">
Pair-program when the engineer wants iteration-time feedback and tight collaboration in the editor. Autonomous PR when the work is well-specified and the engineer reviews the diff at the end rather than during. The framing: pair-program for exploratory work; autonomous PR for queueable bounded tasks.
</CheckQuestion>

<CheckQuestion id="m21-q8" question="8. Devin-class agents: what do they consistently get wrong?">
Long-context tasks where the spec is implicit, anything that requires real-world judgment, complex debugging that needs domain knowledge the codebase does not contain. They handle well-specified medium-sized tasks; they struggle when "make this better" is the actual instruction.
</CheckQuestion>
