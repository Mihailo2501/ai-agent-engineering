Module 02 · Agent patterns

Workflow, agent, or something in between

Module 01 gave you the loop. This module gives you the shapes. Five canonical patterns, when to reach for each, and why the difference between a "workflow" and an "agent" is the most useful taxonomy line you can draw in this whole course.

1 · Workflow vs agent: where control sits

In Module 01 you wrote a single loop: while (stop_reason === 'tool_use') { run_tools(); call_api(); }. That loop is the seed of every multi-step LLM system. The patterns in this module are answers to one design question: who decides what happens next?

Anthropic's Building Effective Agents draws a clean line that this whole module is built around:

  • A workflow orchestrates LLMs and tools through predefined code paths. Your code holds the script. The LLM fills in slots inside that script.
  • An agent dynamically directs its own process. The LLM holds the script. Your code provides the tools and runs the loop, but the LLM picks the next step.

Both are useful. Both have failure modes. The mistake is treating "agent" as a level of sophistication and "workflow" as something simpler. A well-shaped workflow beats a poorly-shaped agent on cost, latency, and predictability for any task that has a stable structure. Reach for an agent when the structure of the task changes from one input to the next, and you cannot enumerate the steps in advance.

The same task, two shapes

Imagine the task: given a company URL, write a one-paragraph briefing for a sales call.

flowchart LR subgraph WF["Workflow shape"] A1[scrape homepage] --> A2[extract product] A2 --> A3[fetch latest news] A3 --> A4[draft paragraph] end subgraph AG["Agent shape"] B1[task: write briefing] --> B2{LLM picks tool} B2 -->|scrape| B3[homepage tool] B2 -->|search| B4[news tool] B2 -->|finish| B5[draft paragraph] B3 --> B2 B4 --> B2 end
Workflow: every step is in your code. Agent: the model picks the next step on every turn until it decides it has enough.

The workflow on the left runs the same four steps every time. Cheap, fast, debuggable, and falls over the moment a homepage uses an unusual layout or the news search returns nothing useful. The agent on the right adapts but costs more per run, takes longer, and is harder to test. Most production systems are not pure either side. They are workflows with one or two agentic steps inside.

The framing that matters. Every pattern in this module is some answer to "how much do I script up front and how much do I let the model decide on the fly?" The implicit dial is the autonomy budget: how many decisions you are willing to hand to the model in exchange for adaptability.

2 · The augmented LLM

Before the patterns, the building block. Anthropic's term for a single LLM call enriched with retrieval, tools, and memory is the augmented LLM. Every pattern in this module is some arrangement of one or more augmented LLMs talking to each other.

Treat this as the unit. When a pattern says "an LLM call," assume that call has retrieval and tools available unless we say otherwise. The patterns differ in how many augmented LLMs are involved, how they are wired together, and who picks the next step.

3 · Prompt chaining

The simplest workflow shape. Prompt chaining decomposes a task into ordered steps, each one an LLM call, with your code carrying outputs forward.

// pseudo-code, not a real API surface
const outline   = await llm('Outline a paragraph about ' + topic);
const draft     = await llm('Write the paragraph from this outline:\n' + outline);
const polished = await llm('Tighten the writing in this draft:\n' + draft);
return polished;

Three calls, three prompts, three small jobs. Each prompt is shorter and more focused than one prompt asking for the whole thing. The model performs better on small jobs than long ones because attention does not have to cover as much in a single turn.

Where chaining wins

  • Quality through narrowing. Splitting a hard task into easy ones reduces error compounding. Each call only has to be right about its slice.
  • Cheap gates between steps. Your code can validate the outline before paying for the draft. If the outline is malformed, retry just that step.
  • Smaller models for cheaper steps. The outline call could go to Haiku and the draft to Opus. You pay the expensive token rate only where you need it.
  • Predictable cost and latency. Every run does N calls. You can budget exactly.

Where chaining breaks

  • Brittleness on edge cases. If step 2 needs information step 1 did not produce, the chain has no way to backtrack. You either widen step 1's prompt (defeating the point) or add error handling (defeating the simplicity).
  • Implicit context loss. Step 3 only sees step 2's output, not step 1's. If you need the original input at step 3 you have to thread it through manually.

Chaining is the right starting shape for almost any structured task. Reach for the more dynamic patterns only after a chain visibly fails.

4 · Routing

Routing is a workflow with a switch at the front. An LLM (or, often, a cheap classifier) reads the input and chooses one of N specialized downstream paths. Each path has its own prompt, tools, and possibly its own model.

const intent = await classify(input, ['refund', 'billing', 'product_question', 'other']);

switch (intent) {
  case 'refund':           return await refundFlow(input);
  case 'billing':          return await billingFlow(input);
  case 'product_question': return await productFlow(input);
  default:                 return await fallbackFlow(input);
}

Routing is what you do when one prompt cannot serve every request well. A refund email and a product question want different system prompts, different tone, different tool sets. Trying to handle both with a single prompt produces flabby output for everyone. Routing lets you specialize without forcing every conversation through the most expensive path.

Sandbox · Build a router

Implement route(input, rules). Given an input string and a list of rule objects shaped { name, keywords }, return the name of the first rule whose keywords appear (case-insensitive) in the input. If no rule matches, return 'fallback'. This is the deterministic half of routing. The "classify" step in production usually replaces this with a cheap LLM call when keywords are not enough.

Two-stage routing

Production routers are usually two stages: a cheap classifier (often Haiku, sometimes a regex or embeddings) decides the lane, and the lane-specific prompt does the actual work. The classifier is small enough that you can run it on every request without breaking your token budget. The lane prompts can be much larger because they only fire on requests that need them.

5 · Parallelization

Run multiple LLM calls at the same time and combine their outputs. Two flavors:

  • Sectioning: split the input into chunks that can be processed independently, then combine the results. Summarizing 50 product reviews by chunking into 5 sets of 10 is a sectioning workload.
  • Voting: run the same prompt multiple times to get diverse outputs, then aggregate (majority vote, average, max, ensemble). Useful when single-call output has too much variance and aggregation across runs gives a more stable answer.

Parallel tool use is parallelization at the model level

The model can also do parallelization for you, inside a single response. Module 01 covered tool use. When the requests are independent, the model emits multiple tool_use blocks in one assistant turn, and your harness should run them concurrently and return all tool_result blocks in a single user-role message. This is built-in parallelization: three enrichment lookups become one round trip instead of three.

By default the model decides whether to parallelize. If you want exactly one tool call at a time, set tool_choice: { type: "auto", disable_parallel_tool_use: true }. The default of letting the model choose is almost always what you want. Sequential tool use is usually a bug, not a feature.

Voting is not magic averaging. Running the same prompt five times and majority-voting only helps when the errors across runs are independent. If the prompt has a structural blind spot, every run will hit it the same way and voting will confidently agree on the wrong answer. Voting raises confidence on tasks the model can already mostly do, not on tasks it fundamentally cannot.

6 · Orchestrator-workers

The orchestrator-worker pattern flips the script. An orchestrator LLM looks at a task, decides what subtasks to spawn, dispatches each to a worker LLM (often with its own tools and context), and synthesizes the results into a final answer.

This is the first pattern that crosses the line from workflow into agent territory. The orchestrator does not know the subtasks in advance. It reads the input and decides them. The same task can produce three subtasks on one input and seven on the next. Your code does not enumerate the steps. The model does.

flowchart TD In["task: research a company"] --> O["orchestrator LLM
plans subtasks"] O -->|"subtask 1: scrape homepage"| W1["worker 1"] O -->|"subtask 2: pull LinkedIn posts"| W2["worker 2"] O -->|"subtask 3: search recent news"| W3["worker 3"] W1 --> S["synthesizer LLM
combines results"] W2 --> S W3 --> S W1 -.->|"empty? re-plan"| O S --> Out["final briefing"]
Orchestrator-worker. The orchestrator decides the cuts. Workers do the work. The synthesizer combines outputs. Dashed edge: optional re-plan path when a worker returns empty.

Step through one orchestration

What makes this work

  • Each worker has a small, clear job. Smaller jobs mean smaller prompts mean fewer model errors per turn.
  • Workers do not share context with each other. Worker 2 cannot accidentally use Worker 1's bad data because it never saw it.
  • The orchestrator can re-plan. If a worker comes back empty, the orchestrator can spawn a different worker with a different approach instead of failing the whole run.

What goes wrong

  • Token cost compounds. Three workers plus an orchestrator plus a synthesizer is five LLM calls for one user turn. Latency adds up. Prompt caching on the shared system prompt is doing real work here.
  • Worker handoff drift. If the orchestrator's subtask description is vague, the worker fills in gaps creatively. Treat each subtask description as a tool description: it is the worker's only briefing.
  • Synthesis is its own job. Three good worker outputs do not combine themselves. The synthesis prompt is often where the orchestrator-worker pattern actually lives or dies.

Sandbox · Implement a tiny orchestrator

You write the dispatcher. The orchestrator has already produced a plan. Implement orchestrate(plan, runWorker): run every worker in parallel, attach each subtask's name to its output, and return an object mapping subtask name to result. runWorker(subtask) is provided. It is async. You must call them concurrently, not one at a time.

7 · Evaluator-optimizer

One LLM generates. Another LLM critiques. The first LLM revises based on the critique. Loop until the critic is satisfied or you hit an iteration cap.

let draft = await generate(task);
for (let i = 0; i < 3; i++) {
  const critique = await evaluate(task, draft);
  if (critique.verdict === 'good') break;
  draft = await revise(task, draft, critique.feedback);
}
return draft;

The evaluator-optimizer pattern is what you reach for when the model can produce almost the right output, but a second pair of eyes catches the kind of error a one-shot prompt misses. Translation polish, code review, prose tightening, structured-output validation. Anywhere the gap between "good enough" and "good" is a critique away.

It works because the evaluator role is genuinely different from the generator role. Asking the same prompt to "write an answer and double-check it" rarely catches its own mistakes. Splitting into two distinct calls with different system prompts gets you the second pass the model could not do in one shot.

The hard part is the evaluator's rubric. A vague evaluator gives vague feedback and the loop just churns. A precise rubric ("the answer must cite at least one source from the provided context, must be under 100 words, and must not invent product names") gets actionable revisions and a real stopping condition.

8 · Subagents in practice

The orchestrator-worker pattern, when you ship it, is what most harnesses call subagents. A subagent is a child LLM call (or a child loop) launched by the parent agent to handle a scoped piece of work. The parent waits, gets a single result back, and continues.

Claude Code's Agent tool is the canonical example. The parent emits a tool_use block calling the Agent tool with a description and a prompt. Claude Code spawns a fresh agent loop with its own context window. That child runs to completion, returns a single text result, and the parent continues with that result as a tool_result block.

{
  "name": "Agent",
  "input": {
    "subagent_type": "Explore",
    "description": "Find auth middleware",
    "prompt": "Find files defining session token validation. Report paths and line ranges. Under 200 words."
  }
}

Why subagents exist

  • Context isolation. The parent does not pollute its own context with raw search results. The subagent reads ten files, summarizes, and returns one paragraph. The parent sees only the paragraph.
  • Specialized prompts. An "Explore" agent is read-only and prompted for fast lookups. A "Plan" agent is prompted to produce an implementation plan. The parent picks the right shape for the job.
  • Parallelism. The parent can emit two or three Agent tool_use blocks in a single turn and the harness runs them concurrently. Three independent searches finish in the time of one.

What you give up

  • Subagents do not see parent context. Files the parent has read, prior tool results, the conversation, none of it. The prompt you pass is the entire briefing.
  • Cost. Each subagent is a full agent loop with its own input, system prompt, and tool definitions. The parent pays for itself plus the children.
  • Debugging surface area. A subagent's reasoning is invisible from the parent's transcript. If the child returns a wrong answer, you have to inspect its loop separately.

This is the same tradeoff you saw in the orchestrator-worker section, just packaged. Most harnesses (Claude Code, Codex, the Claude Agent SDK) ship their version of this primitive. Module 05 unpacks the harness side. Module 07 covers the managed-platform side.

9 · Choosing a pattern

The decision is rarely "which one is best." It is "which one is the smallest shape that solves my task without breaking when the input shifts." Climb the ladder only when the simpler shape visibly fails on real inputs.

If your task is...Start withClimb to
One transformation, stable inputSingle augmented LLM callChaining when the task gets too long for one prompt
Multiple transformations, fixed orderPrompt chainingRouting if the first step branches by type
One of N stable input typesRoutingParallelization inside each lane if needed
Independent subproblems, fixed shapeParallelization (sectioning)Orchestrator-workers when the shape varies
Variable subproblems, plan unknown up frontOrchestrator-workersFull agent loop when even the orchestrator's plan needs to react
Output close to right but needs polishEvaluator-optimizerStays here. This is its lane.
Open-ended, many turns, environment feedbackAgent loop with toolsSubagents (see §8) only when one agent's context is the bottleneck

The autonomy budget, made concrete

Every step you hand to the model costs you predictability and buys you adaptability. A one-shot prompt is fully predictable and fully rigid. A full agent loop with twenty tool calls is fully adaptive and barely auditable. Most production systems sit in the middle: a workflow that wraps one or two agent steps, where the dynamic decisions are bounded by the static structure around them.

The rule of thumb: script the parts that never change. Hand the parts that vary by input to the model. A briefing pipeline can hard-code "scrape, then enrich, then write" but use an agent loop inside the enrichment step because what to enrich varies by company. The shape of the system stays predictable. The flexibility lives where it has to.

Sandbox · Pick the pattern

You are the architect. For each task description, return the right pattern name. Valid returns: "chaining", "routing", "parallel", "orchestrator", "evaluator", "agent". The verdicts in this sandbox are normative, not aesthetic. They reflect the table above.

Where Potter sits in this taxonomy

Potter is a tool surface. It is not an agent. Internally, every Potter function is a workflow: scripted code paths that hit one or more enrichment providers, normalize the response, and return a structured result. There is exactly one exception. potter_browser_act is the one tool with an LLM inside it (Stagehand drives a browser action with a model in the loop). Everywhere else, Potter is predefined paths.

The agent lives outside Potter, in the harness. Claude Code (or any other MCP-speaking harness) holds the agent loop. It picks tools. It decides when to stop. Potter just executes whatever tool the harness asks for and returns a tool_result.

This is a deliberate split that maps onto every pattern in this module:

  • When a Potter user runs a research query, the harness is the agent. It might call potter_research_company, then potter_linkedin_posts, then potter_web_search, deciding the next call based on what came back.
  • Inside potter_research_company, Potter runs a chain: hit Proxycurl, fall back to Apify if needed, normalize, return.
  • Inside potter_find_decision_maker, Potter runs a parallel shape: query several providers concurrently, pick the best result.
  • Inside potter_browser_act, there is a small agent: Stagehand interprets a natural-language instruction and decides which DOM actions to take.

The lesson is the locus of control. The agent loop runs in the harness because that is the only place that knows what the user wants. Potter's job is to make tools available and run them deterministically when called. The patterns in this module are how you decide where to put each loop. Potter's design is one valid answer.

Glossary

workflow
A system where LLMs and tools are orchestrated through predefined code paths. Your code holds the script.
agent
A system where the LLM dynamically directs its own process and tool usage. The model holds the script.
augmented LLM
A single LLM call enriched with retrieval, tools, and memory. The atomic unit every pattern in this module is built from.
prompt chaining
Workflow pattern: decompose a task into ordered LLM calls, each one a smaller, more focused job than the original.
routing
Workflow pattern: classify the input and dispatch to one of N specialized downstream paths.
parallelization
Running multiple LLM calls concurrently and combining outputs. Two flavors: sectioning (different chunks) and voting (same prompt, multiple runs).
sectioning
Parallelization variant: split the input into independent chunks, run each in parallel, combine the results.
voting
Parallelization variant: run the same prompt N times to get diverse outputs, then aggregate (majority vote, average, ensemble). The aggregation step reduces variance. Does not help with structural blind spots.
orchestrator-worker
Pattern: an orchestrator LLM plans subtasks at runtime, dispatches each to a worker LLM, and synthesizes the worker outputs.
evaluator-optimizer
Pattern: a generator LLM produces output, an evaluator LLM critiques it, the generator revises. Loops until the evaluator passes or a cap is hit.
subagent
A child agent loop launched by the parent. Has its own context window and tools, returns a single result, does not see parent context.
autonomy budget
The implicit dial in every pattern: how many decisions you hand to the model in exchange for adaptability. Higher autonomy means more flexibility and less predictability.

Check yourself

Answer these in your own words before the next session. Click each to see how I would answer.

1. What is the load-bearing distinction between a workflow and an agent? Why does it matter for system design?
show answer
A workflow orchestrates LLMs and tools through predefined code paths your application controls. An agent dynamically directs its own process: the model picks the next step at each turn. The distinction matters because agents pay extra cost and latency for adaptability, and you only want to pay that when the structure of the task actually varies by input. A workflow with one or two agent steps inside is the most common production shape.
2. The model returns three tool_use blocks in one response. What does that mean and what should the harness do?
show answer
The model is asking for three tool calls and judged them independent enough to run in one round trip. The harness should execute all three concurrently (Promise.all, asyncio.gather, the equivalent in your runtime) and return all three tool_result blocks in a single user-role message. Returning them sequentially or in separate user messages defeats the parallelism the model already opted into. Default behavior leaves parallel tool use enabled. Disable it only with tool_choice.disable_parallel_tool_use: true, and only when sequential ordering matters for correctness.
3. When does prompt chaining outperform a single bigger prompt, and when does it underperform?
show answer
Chaining wins when the task decomposes cleanly into smaller jobs, each easier than the whole. The model performs better on small, focused prompts; you can validate intermediate outputs cheaply; you can use cheaper models on cheaper steps. Chaining loses when later steps need information you did not anticipate at earlier steps, when input context has to thread through multiple hops, or when the decomposition itself is the variable thing (in which case orchestrator-workers fits better because the orchestrator picks the cuts at runtime).
4. The two flavors of parallelization are sectioning and voting. What is each one good for, and what does voting fail to fix?
show answer
Sectioning splits the input into independent chunks and processes each in parallel: summarizing fifty reviews by chunking into five batches of ten. Voting runs the same prompt multiple times and aggregates: majority vote, average, ensemble. Sectioning is for throughput and context-window relief. Voting is for variance reduction. Voting does not fix structural blind spots: if every run gets the same wrong answer for the same reason, the majority will confidently agree on the wrong answer. Voting only helps when errors across runs are roughly independent.
5. In the orchestrator-worker pattern, why does the synthesis step usually need its own dedicated prompt rather than letting the orchestrator do it inline?
show answer
Three good worker outputs do not combine themselves. Synthesis is a different cognitive job from planning: it has to read multiple inputs, resolve conflicts, drop irrelevant detail, and produce one coherent answer in the shape the user expects. Asking the orchestrator to "plan, dispatch, and also write the final answer" overloads its context with raw worker outputs and dilutes the planning prompt. A dedicated synthesis prompt sees only the worker outputs and the original goal, with a clear job to do. It usually goes to the same model the orchestrator uses, but with a different system prompt.
6. A subagent in Claude Code is launched via the Agent tool. What does the parent give up by delegating, and what does it gain?
show answer
Gives up: visibility into the child's reasoning trace (only the final result comes back), the ability to share parent context (the child cannot see files the parent read or prior tool results), and a real cost line item (each subagent is its own agent loop with its own input + system prompt + tool definitions). Gains: context isolation (parent does not get polluted with raw search results), specialization (Explore is read-only and tuned for fast lookups, Plan is tuned for design proposals), and parallelism (multiple Agent tool_use blocks in one turn run concurrently). Use a subagent when the work has a clean boundary and you want a one-paragraph result, not when you need the parent to participate in the reasoning.
7. You are designing a system that takes a customer support email and produces a reply. The email type varies (refund, billing, product question, other). Each type needs a different tone, different RAG sources, and different escalation rules. What pattern fits, and where does it sit on the autonomy budget?
show answer
Routing. A cheap classifier (Haiku, embeddings, or a regex if the keywords are unambiguous) reads the email and tags one of four types. Each lane has its own system prompt, its own retrieval scope, and its own escalation logic. This sits low on the autonomy budget: the structure of the system is fully scripted, and the only "agent" decision is the lane choice. You get specialization without paying for full agent autonomy on every email. Climb to a real agent loop only inside lanes where the resolution genuinely varies turn by turn (e.g., a multi-turn refund conversation that has to look up order history, send confirmations, and update tickets).
8. Why is "always reach for the most flexible pattern" the wrong default, and what is the right one?
show answer
Flexibility is not free. Each step of autonomy you grant the model trades predictability and cost for adaptability. A full agent loop on a task that could be a chain wastes tokens, adds latency, makes evaluation harder, and surfaces a class of failure modes (infinite tool loops, drift, runaway iteration) that scripted code does not have. The right default: start with the smallest pattern that fits, and climb only when the simpler shape demonstrably fails on real inputs. Most "we need an agent" tasks are actually chains, routes, or orchestrator-workers in disguise.
