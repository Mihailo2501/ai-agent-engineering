<Section id="what-is-llm" num={1} title="What an LLM is, in three minutes">

A large language model is a statistical machine trained to predict the next token given a sequence of previous tokens. <GLink slug="token">Tokens</GLink> are sub-word fragments. The word "Paris" is roughly one token, "Subotica" is two or three. The model has no concept of facts, no concept of time, no internal database. It has a very large set of weights that, given a sequence of tokens, produce a probability distribution over what token comes next.

That is the entire mechanism. Everything else (reasoning, instruction-following, tool use, agentic behavior) is an emergent property of training that machine on enormous amounts of text and giving it a particular shape of input. The model alone is not an agent. It is a very fast next-token predictor wrapped in an HTTP API.

Three properties matter for everything in this course:

- <GLink slug="stateless-api"><strong>Stateless.</strong></GLink> The model has no memory of previous calls. Every API request starts from zero. Whatever the model "knows" about your conversation is whatever you put in the messages array right now.
- **Probabilistic.** Same input does not produce identical output. The same prompt called twice can yield slightly different responses unless you pin sampling parameters carefully.
- **Bounded.** The model has a <GLink slug="context-window">context window</GLink>: a maximum number of tokens it can attend to in a single call. Hit the limit and the API rejects the request.

<Callout kind="info" title="Why this framing matters.">
Every architectural decision in agent design is a workaround for one of those three properties. State management, deduplication, retry logic, compaction, prompt caching, multi-agent patterns, all of it exists because the underlying model is stateless, probabilistic, and bounded.
</Callout>

</Section>

<Section id="api-shape" num={2} title="The shape of an API call">

An LLM API call is a single HTTP POST with a JSON body. You send a list of messages, you get back a response. That is the entire surface. There is no session, no memory, no state on the server side. If you want a conversation, you keep the message history yourself and resend it every time.

<CodeBlock lang="json">{`// POST https://api.anthropic.com/v1/messages
{
  "model": "claude-opus-4-7",
  "max_tokens": 1024,
  "system": "You are a helpful research assistant.",
  "messages": [
    { "role": "user", "content": "What's the weather in Paris?" }
  ]
}`}</CodeBlock>

The response comes back as a similar JSON object: a list of content blocks, a <GLink slug="stop-reason"><code>stop_reason</code></GLink> telling you why the model stopped, and a `usage` object with token counts. That second field is more important than people realize. It is how the <GLink slug="harness">harness</GLink> decides what to do next.

<CodeBlock lang="json">{`{
  "id": "msg_01ABC...",
  "type": "message",
  "role": "assistant",
  "content": [
    { "type": "text", "text": "I don't have real-time weather data..." }
  ],
  "stop_reason": "end_turn",
  "usage": { "input_tokens": 42, "output_tokens": 28 }
}`}</CodeBlock>

### Request fields you will touch every day

- `model`: which Claude variant. Affects cost, speed, capability.
- `max_tokens`: hard cap on output length. Required.
- `system`: the system prompt. Persistent instructions for the model.
- `messages`: the conversation history.
- `tools`: tool definitions the model can invoke (covered in §5).
- `temperature`: randomness, 0 to 1 (covered in §4).
- `stream`: boolean, returns SSE instead of single response (covered in §8).

</Section>

<Section id="roles-blocks" num={3} title="Roles and content blocks">

Every message has a `role`. There are exactly three real roles:

- **system**: instructions from the developer. Sets behavior, persona, constraints. Lives in its own top-level field, not in the messages array.
- **user**: the human (or your program) speaking to the model. Whatever you want a response to.
- **assistant**: the model's previous turns. You include these when continuing a conversation so the model has its own history.

And inside a message, the `content` field is either a string or an array of **content blocks**. The block types you will see:

- `text`: plain text.
- <GLink slug="tool-blocks"><code>tool_use</code></GLink>: emitted by the assistant when it wants to call a tool (§5).
- `tool_result`: sent back by you (in a user-role message) with the tool's output.
- `image`: vision input.
- `thinking`: extended reasoning blocks (newer models).

Five shapes. Every agent in existence is built from these in some order.

<Section id="sampling" num={4} title="How the model decides">

The model produces a probability distribution over the next token at every step. Sampling is the rule for picking one token from that distribution. The two parameters that matter most:

- <GLink slug="temperature"><strong>temperature</strong></GLink>: scales the distribution. Low temperature (near 0) makes the highest-probability token nearly certain. High temperature (1+) flattens the distribution and produces more variety.
- **top_p**: nucleus sampling. Only consider tokens whose cumulative probability reaches `p`. Cuts off the long tail of low-probability options.

For agent design this matters because **same input does not give same output**. If your harness retries a failed tool call, the model can pick a different tool the second time. If you want reproducibility for evaluation, set `temperature: 0` and pin everything else. For creative work, raise temperature. For tool selection in production, low temperature is usually right. You want the model to pick the same tool every time it sees the same situation.

<Callout kind="warn" title="The trap.">
Even with <code>temperature: 0</code>, identical output is not guaranteed across model versions or across batches. Treat reproducibility as best-effort, not a guarantee. Eval suites should account for variance, not assume it away.
</Callout>

</Section>

### The tool-use lifecycle

When the model decides to call a tool, the response looks like this:

<CodeBlock lang="json">{`{
  "content": [
    { "type": "text", "text": "Let me check that for you." },
    {
      "type": "tool_use",
      "id": "tu_01XYZ...",
      "name": "get_weather",
      "input": { "city": "Paris" }
    }
  ],
  "stop_reason": "tool_use"
}`}</CodeBlock>

The harness inspects the response, executes `get_weather({ city: "Paris" })` in your code, and sends the result back as a **user-role** message containing a tool_result block:

<CodeBlock lang="json">{`{
  "role": "user",
  "content": [
    {
      "type": "tool_result",
      "tool_use_id": "tu_01XYZ...",
      "content": "18°C, partly cloudy"
    }
  ]
}`}</CodeBlock>

The `tool_use_id` links the result back to the request. The model then continues generating, this time with the tool's output in its context.

<Callout kind="warn" title="Tool descriptions are the most important strings you will write.">
The model picks which tool to call based almost entirely on the description. A vague description means missed or wrong tool calls. Treat descriptions as prompts, not docs.
</Callout>

### Parallel tool use

The model can emit multiple `tool_use` blocks in a single response. The harness should run them <GLink slug="parallel-tool-use">in parallel</GLink> where possible and return all results as content blocks in a single user-role message. This is a major performance win. Three independent enrichment calls take one round-trip instead of three.

### Tool errors

If your tool throws, send the error back as a tool_result with `"is_error": true` and a string description of what failed. The model can then decide to retry, try a different tool, or surface the error to the user. Do not swallow tool errors silently. The model needs the signal.

<Section id="agentic-loop" num={6} title="The agentic loop">

One API call gets you one turn. An *agentic loop* is what happens when the model emits a tool_use, your code runs the tool, and you call the API again with the result appended. The loop runs until the model stops asking for more tools.

The harness loop is driven by one field: `stop_reason`. It tells you why the model stopped generating, and dictates what to do next.

### What can go wrong inside the loop

- **Infinite tool loops.** The model keeps asking for the same tool with slightly different arguments. Cap iterations and break with an explicit error message.
- **max_tokens during tool_use.** The model started a tool call but hit max_tokens mid-JSON. The tool_use block is malformed. Either raise max_tokens or detect and retry.
- **Tool selection drift.** With temperature above 0, the model can pick different tools across runs for the same input. Pin temperature to 0 if you need determinism.
- **Context exhaustion.** Each loop iteration adds messages. After 20 iterations of large tool results, you can hit the context limit. Compaction or subagents address this.

</Section>

<Section id="context-window" num={7} title="The context window">

The messages array has a hard limit measured in tokens. Claude Opus 4.7 with the 1M context tier holds roughly 1,000,000 tokens. The standard tier holds 200,000. Once you hit the cap the API rejects the request. That ceiling drives almost every architectural decision in agent design: when to compact, when to summarize, when to spawn a subagent, what to put in the system prompt versus the user message.

A practical rule: assume English averages four characters per token. A 200K-token window is roughly 800K characters, which is roughly 150,000 words, which is two and a half novels. Big, but not infinite. One badly-shaped tool that returns a 50K-token JSON blob will eat 25% of it in a single call.

<Callout kind="note" title="Why responses get capped.">
A single LinkedIn profile dump can run 8K+ tokens. Three of those and you have spent a non-trivial chunk of the agent's working memory on raw enrichment data. Truncation markers are not aesthetic. They are budget management.
</Callout>

### Prompt caching, the optimization you should know exists

If your system prompt and tool definitions do not change between calls, the API will cache that prefix and only charge full price for the new bits. Properly used, <GLink slug="prompt-caching">prompt caching</GLink> cuts cost by 80 to 90% on agent loops because the system prompt and tool list are sent on every iteration. You add a `cache_control: { type: "ephemeral" }` marker at the boundary of cacheable content, and the API takes care of the rest. Cache TTL is roughly 5 minutes, refreshed on each hit.

For an agent loop with a 10K-token system prompt and 20 iterations, prompt caching turns 200K input tokens into roughly 20K of cached read plus the new bits. Same accuracy, fraction of the cost.

</Section>

<Section id="streaming" num={8} title="Streaming">

By default, the API holds the response until the model finishes generating. With `stream: true`, the API returns <GLink slug="sse">Server-Sent Events</GLink> as the model produces them. The events look like:

- `message_start`: fires once. Contains the message envelope minus content.
- `content_block_start`: a new content block is opening (text, tool_use, etc.).
- `content_block_delta`: incremental content for the open block. For text, a partial string. For tool_use, partial JSON.
- `content_block_stop`: the block is complete.
- `message_delta`: top-level fields update (stop_reason, usage).
- `message_stop`: the response is complete.

Streaming matters for three reasons: **UX** (users see tokens as they arrive instead of waiting), **early cancellation** (you can stop generation if the user navigates away), and **partial tool inputs** (you can start preparing a tool call before the model finishes generating its arguments).

For agentic loops you usually want streaming on. Latency on a long generation can be 30+ seconds. Users will not wait 30 seconds staring at a blank screen.

</Section>

<Section id="first-agent" num={9} title="Your first 20-line agent">

Putting it all together. This is a complete, working agent. Read the comments. Every line is doing something the previous sections explained.

<CodeBlock lang="ts">{`import Anthropic from '@anthropic-ai/sdk';

const client = new Anthropic();   // reads ANTHROPIC_API_KEY from env
const messages = [{ role: 'user', content: "What's the weather in Paris?" }];

const tools = [{
  name: 'get_weather',
  description: 'Look up current weather for a city. Returns temp in C and conditions.',
  input_schema: {
    type: 'object',
    properties: { city: { type: 'string', description: 'City name' } },
    required: ['city']
  }
}];

async function runTool(name, input) {
  if (name === 'get_weather') return \`18°C, partly cloudy in \${input.city}\`;
  throw new Error(\`unknown tool: \${name}\`);
}

while (true) {
  const resp = await client.messages.create({
    model: 'claude-opus-4-7', max_tokens: 1024, tools, messages
  });
  messages.push({ role: 'assistant', content: resp.content });

  if (resp.stop_reason === 'end_turn') {
    const text = resp.content.find(b => b.type === 'text')?.text;
    console.log(text);
    break;
  }

  const toolResults = [];
  for (const block of resp.content) {
    if (block.type === 'tool_use') {
      const output = await runTool(block.name, block.input);
      toolResults.push({ type: 'tool_result', tool_use_id: block.id, content: output });
    }
  }
  messages.push({ role: 'user', content: toolResults });
}`}</CodeBlock>

<Section id="check" title="Check yourself">

<CheckQuestion id="m01-q1" question={'1. The API is described as "stateless". What does that mean in practice, and what is its consequence for how you write agent code?'}>
The Anthropic API does not store any conversation state between requests. Every call is independent. You send the full messages array, you get a response, the server forgets you exist. The consequence: memory is a client-side problem. If your agent needs context from earlier turns, your harness must store that conversation history somewhere (in memory, in a database, in a file) and resend it on every call.
</CheckQuestion>

<CheckQuestion id="m01-q2" question={'2. What does stop_reason: "tool_use" mean, and what happens next?'}>
The model paused generation because it wants to call one or more tools. The harness reads the tool_use block(s) from the response, executes the named tool with the provided arguments, appends a user-role message containing tool_result blocks (one per tool_use, linked by tool_use_id), and calls the API again with the updated messages array. The loop repeats until stop_reason is end_turn.
</CheckQuestion>

<CheckQuestion id="m01-q3" question="3. Why is the description field on a tool definition load-bearing, and what is the failure mode of getting it wrong?">
The model picks which tool to call based almost entirely on the natural-language description and parameter docstrings. A vague description means the model either skips the tool when it should use it, or picks the wrong tool when two have overlapping descriptions. Tool descriptions are prompts, not docs. They are the model's only interface to your API surface.
</CheckQuestion>

<CheckQuestion id="m01-q4" question="4. What does the harness add on top of the API that the API does not provide?">
The loop, the tool registry, the tool execution, conversation persistence, permission prompts, file system access, subagent management, audit logs, and any UI. The API gives you one model call. The harness wraps that call in the loop that makes "agent" behavior possible. Claude Code, Codex, Cursor, n8n, and a 20-line custom script are all variations of the same basic harness shape.
</CheckQuestion>

<CheckQuestion id="m01-q5" question="5. Why does temperature matter for agent design, and what setting would you pick for tool selection in production?">
Temperature controls how random the model's token sampling is. With temperature above 0, the same input can produce different outputs across runs, including different tool selections for the same situation. For production tool selection, you usually want temperature 0 or very low. Identical inputs should pick the same tool to keep behavior predictable, debuggable, and testable. Raise temperature for creative or exploratory tasks where variety is the goal.
</CheckQuestion>

<CheckQuestion id="m01-q6" question="6. In the Potter stack diagram, where does the agentic loop run? Why is that the right place for it?">
The loop runs in Claude Code, the harness. Putting it there keeps Potter stateless and composable. Potter ships tools, Claude Code orchestrates. If the loop lived inside Potter you would have a model deciding tools inside another model's loop, doubling cost, fragmenting context, and tying Potter to a specific LLM. The MCP boundary exists precisely to keep these layers separate, and it lets Potter work in any MCP-speaking harness.
</CheckQuestion>

<CheckQuestion id="m01-q7" question="7. What is prompt caching and why does it matter for a long agent loop?">
Prompt caching lets the API cache a prefix of repeated requests, typically the system prompt and tool definitions, and charge a fraction of the normal price for cached reads. For an agent loop that calls the API 20 times in a session with the same 10K-token system prompt, caching cuts that 200K of repeated input tokens to roughly 20K cached reads. Same accuracy, around 80 to 90% cost reduction. You opt in by adding a cache_control marker at the boundary of cacheable content.
</CheckQuestion>

<CheckQuestion id="m01-q8" question="8. Name three things the 20-line agent skeleton lacks and where each capability comes from in production.">
Examples: file system access (added by Claude Code or a custom toolset), permission prompts (added by Claude Code's UI layer), subagents (Module 02 patterns), streaming (change <code>create</code> to <code>stream</code> and consume events), caching (cache_control on system + tools), iteration cap (manual for-loop with break to prevent infinite tool loops), error handling (try/catch around tool execution, surfaced as is_error tool_results), observability (logging every call's usage and stop_reason, covered in Module 12).
</CheckQuestion>
