<Section id="prompting" num={1} title="Why tool design is prompt engineering">

A tool definition is a prompt with a schema attached. The model picks tools based almost entirely on the description. Module 01 made the point at small scale: with five tools you can mostly get away with terse descriptions. At 25 tools, terse becomes ambiguous; at 50, ambiguous becomes unusable.

The failure mode at scale is not "model invents a tool" or "model picks something obviously wrong." It is **selection drift**: small ambiguities between two similar tools cause the model to pick inconsistently across runs. One time it picks `lookup_company`; next time on the same input it picks `search_web`; both work, neither is *the* right call. Eval suites suffer; logs become hard to read; collaborators wonder why the agent behaves differently for them than for you.

The fix is discipline rather than bigger descriptions. Naming, schema design, error shapes, gating, retrieval: each one absorbs a class of selection drift. The rest of this module is the discipline.

<Section id="naming" num={2} title="Naming conventions that scale">

Names matter more at scale than at small. With five tools the model recalls each description in full; with fifty, it leans on patterns in names to disambiguate.

The conventions worth following:

- **snake_case.** Avoid camelCase and kebab-case in tool names. Models tokenize snake_case more cleanly and use the underscores as boundaries.
- **Namespace prefix.** Group related tools with a leading namespace. `potter_research_*`, `potter_browser_*`, `potter_linkedin_*`, `potter_web_*`. The namespace tells the model which family of tools to consider before reading descriptions.
- **Verb-then-noun.** `lookup_company`, `send_email`, `extract_text`. Verbs first match the model's intent inference; noun-first names (`company_lookup`) feel less natural and degrade selection.
- **Avoid abbreviations.** `get_user_email_address` beats `get_user_email_addr`. The model occasionally mis-recalls abbreviated names (especially across model versions); full words are stable.
- **Disambiguate similar tools by suffix.** `lookup_company` and `lookup_company_lite` are clearer than `lookup_company` and `quick_company_lookup`.

The smell test: read the tool name out loud as a verb phrase. "Lookup company" is fine. "Company lookup" is fine for a noun. "Lkup co" is not fine.

</Section>

<Section id="descriptions" num={3} title="Descriptions as prompts">

The description is the most load-bearing string in your tool definition. Treat it as a prompt, not as documentation.

The shape that wins:

1. **What the tool does, in one sentence.** "Look up a company by domain. Returns name, size, industry, headquarters."
2. **When to call this.** "Use this when the user mentions a company name or domain and asks about who they are, what they do, or how big they are."
3. **When NOT to call this.** "Do not call for individuals; lookup_person covers that."
4. **What the output looks like.** Critical when the agent needs to chain calls: "Returns { name, size, industry, hq } as a JSON object."

A vague description has none of these; a great description has all four. The "when NOT to call" beat is the one most people skip, and it is the one that fixes most selection drift between tools with overlapping coverage.

<Section id="schemas" num={4} title="Schema design tricks">

The <GLink slug="schema">schema</GLink> describes a tool's input arguments. JSON Schema; both Anthropic and MCP use it. The tricks worth knowing:

- **Required vs optional.** `required` is a list of keys the model must fill. The model can still skip required fields under stress; your tool should validate before calling external APIs.
- **Enums for closed sets.** When a field has a fixed list of values, use `enum: ["small", "medium", "large"]`. The model attends to enums more reliably than to free-form strings with "must be one of small, medium, large" tucked in the description.
- **Defaults via description, not schema.** JSON Schema's `default` is not enforced by the API. Document the default in the field description ("If omitted, defaults to 10.") so the model knows.
- **Conditional fields via `oneOf`.** When two fields are mutually exclusive (provide either `domain` or `name`, not both), `oneOf` makes the constraint explicit. The model reads it; the API validates it.
- **Max array lengths.** When a field is an array (`tags: string[]`), set `maxItems`. Prevents the model from generating 100-element arrays when 5 was the intent.
- **Nested shapes via `$ref`.** When two tools share a sub-schema (a `Person` object, an `Address` object), reference it instead of duplicating. Easier to evolve.

A worked example of a richer schema:

<CodeBlock lang="json">{`{
  "name": "lookup_company",
  "description": "Look up a company by domain or name. Returns size, industry, headquarters.",
  "input_schema": {
    "type": "object",
    "oneOf": [
      { "required": ["domain"] },
      { "required": ["name"] }
    ],
    "properties": {
      "domain": {
        "type": "string",
        "description": "Company website domain, e.g. acme.com. Provide this OR name, not both."
      },
      "name": {
        "type": "string",
        "description": "Company name, e.g. Acme Corp. Use only when domain is unknown."
      },
      "tier": {
        "type": "string",
        "enum": ["lite", "full"],
        "description": "Lookup tier. lite is fast and cheap; full triggers Proxycurl. If omitted, defaults to lite."
      }
    }
  }
}`}</CodeBlock>

</Section>

<Section id="errors" num={5} title="Error response patterns the model can recover from">

Tool errors should return structured data the model can introspect. Free-text errors leave recovery to the model's improvisation; structured errors let the agent pattern-match on the error kind and decide what to do.

The pattern that holds across production agents:

<CodeBlock lang="json">{`{
  "is_error": true,
  "kind": "rate_limited",
  "message": "Proxycurl rate limit exceeded for this minute.",
  "retry_hint": "retry-after-30"
}`}</CodeBlock>

The fields:

- **`is_error: true`.** The flag the API and the model both look at.
- **`kind`.** A short, machine-readable category. Common kinds: `rate_limited`, `not_found`, `auth_failed`, `network`, `validation`, `internal`.
- **`message`.** Human-readable text for logs and the model's internal reasoning.
- **`retry_hint`.** A recommendation: `retry-after-N`, `retry`, `fix-input`, `abort`. The agent can pattern-match this directly.

Why structured errors win:

- The agent can decide retry policy without re-reading prose. `retry-after-30` is not a sentence; it is an instruction.
- Eval suites can assert on error kinds without flaky string matching.
- Logs become structured; observability dashboards work.
- When a tool fails for a new reason, you add a new `kind` instead of editing prose all over the codebase.

<Section id="gating" num={6} title="Gating: per-user, per-context, per-time">

Not every tool should be auto-approved. <GLink slug="tool-gating">Tool gating</GLink> is the policy layer that decides whether a tool runs immediately, asks for confirmation, or refuses.

The gating dimensions worth thinking about:

- **By tool.** Read-only tools commonly auto-approved; writes, network calls, expensive operations gated.
- **By user.** A senior engineer's session might allowlist destructive operations; a customer-support agent's session might not.
- **By context.** "During business hours" gating, "in production" gating, "outside dev environment" gating.
- **By cost.** Tools that consume billed external API credits (Proxycurl, Apify) should confirm before running, especially in batch.
- **By blast radius.** Tools that delete or send messages should confirm; tools that read should not.

The pattern most production agents converge on: declare the gating policy inline with the tool definition, not in a global config that the model never sees. Knowing the policy (or at least its existence) helps the model frame its own intent: "this is a confirm-before-run tool, so I should ask the user first."

</Section>

<Section id="semantic-search" num={7} title="Semantic search across a large tool catalog">

When the catalog grows past 30 to 50 tools, the model starts missing relevant ones because each tool's description competes for attention. The fix is retrieval before selection: pre-filter the catalog by relevance to the current query, then show the model only the top-K candidates.

The shape:

1. Embed all tool descriptions once (offline, cached).
2. On each query, embed the query and find the top-K most similar tools.
3. Pass only those K to the model.
4. Model picks from K instead of the full catalog.

For small catalogs (under 30 tools), full catalog plus solid descriptions is enough. For large catalogs, pre-filtering is the difference between "the agent finds the right tool" and "the agent picks one of three semi-right tools at random."

A simple substring-based version (no embeddings, just keyword overlap) handles the small-but-growing case: count query-word matches in each tool's name and description, sort, return top-K. It works surprisingly well and is the right starting point before reaching for embeddings.

<Section id="plan-then-execute" num={8} title="Have the model describe its plan before calling">

A pattern that often beats raw tool selection at scale: ask the model to describe its plan first, then execute. The plan acts as a retrieval step the model does cognitively, narrowing the tool space before any actual tool_use block fires.

The shape:

1. System prompt instructs the model: "Before calling tools, write a 2-3 sentence plan describing what you intend to do and which tools you will use."
2. Model outputs the plan as a text block, then emits tool_use blocks.
3. The plan acts both as observability (you see what the model thought) and as constraint (the model picks tools that match its own stated plan).

When this helps:

- Multi-step tasks where wrong early choices cascade.
- Catalogs where several tools have overlapping coverage.
- Eval debugging: the plan tells you why the model picked the wrong tool.

When it does not help:

- Latency-sensitive paths (the plan adds tokens).
- Single-tool tasks (the model would emit the right call without the plan anyway).
- High-volume cheap calls (plan tokens add up at scale).

The cost is roughly 50 to 200 extra tokens per turn. Worth it for hard tasks; overhead for easy ones.

</Section>

<Section id="versioning" num={9} title="Tool versioning and deprecation">

Tools evolve. The signature changes; the output shape changes; a better tool ships. The discipline:

- **Ship v2 alongside v1.** New name (`lookup_company_v2` or just `lookup_company` with an updated description noting the change). Existing prompts keep using v1; new prompts can adopt v2.
- **Mark v1 deprecated in its description.** "DEPRECATED: use lookup_company_v2 instead. This tool will be removed on 2026-09-01." The model reads the description and prefers the new tool.
- **Communicate the migration.** README, CHANGELOG, prompt-update guidance for whoever uses the tool.
- **Keep v1 working through the grace period.** Removing a tool that other code depends on without warning breaks production agents silently.

The anti-pattern: silently changing v1's behavior or output shape. Existing prompts assume the old shape; downstream code parses results based on it. Hidden behavior changes break agents in ways that are very hard to diagnose.

</Section>

<Section id="anti-patterns" num={10} title="Anti-patterns: what to never do">

A short list of things that hurt every time:

- **Two tools that do the same job under different names.** The model picks inconsistently. Pick one, deprecate the other.
- **Returning huge JSON blobs as tool results.** A 50K-token result inflates context and corrupts attention. Truncate, summarize, or redesign the tool to return only what the agent needs.
- **Letting the model guess output shapes.** Document the output in the description. The agent should not have to introspect a tool result to find what fields exist.
- **Free-text errors.** "Failed: something went wrong" gives the agent nothing to recover from. Structured errors with a kind.
- **Sharing tool descriptions across modes.** A tool whose description says "this can do X or Y depending on context" splits attention. Two tools with two clear descriptions usually win.
- **Hedging in descriptions.** "Maybe useful for company lookup": the model reads "maybe" and picks something else.
- **Silent rate limiting.** Tool waits 30 seconds then returns success. The agent has no idea anything happened. Either return a structured `rate_limited` error and let the agent retry, or expose a `wait` parameter so the wait is visible.

<Section id="check" title="Check yourself">

<CheckQuestion id="m10-q1" question="1. Why is naming convention more important at 25+ tools than at 5?">
At small scale the model recalls each description fully and disambiguates from content. At large scale, descriptions compete for attention and the model leans on names as a first-pass filter. Namespace prefixes (potter_research_*, potter_browser_*) let the model consider only a relevant subset before reading descriptions in detail.
</CheckQuestion>

<CheckQuestion id="m10-q2" question="2. What is the most load-bearing string in a tool definition?">
The description, by far. The model reads it to decide whether to invoke at all and which arguments to fill. Vague descriptions get skipped; precise descriptions get picked. Treat the description as a prompt: what the tool does, when to call, when NOT to call, what the output looks like.
</CheckQuestion>

<CheckQuestion id="m10-q3" question='3. Why include "when NOT to call this" in a description?'>
Tools with overlapping coverage cause selection drift: the model picks one inconsistently across runs because the descriptions both fit. The negative description is the disambiguator. "Use lookup_company; do not call for individuals (lookup_person covers that)" tells the model where the boundary is.
</CheckQuestion>

<CheckQuestion id="m10-q4" question="4. What is a structured error and why prefer it over free text?">
JSON with kind, message, and retry_hint. The model can pattern-match on kind to decide retry vs fix-input vs abort without re-reading prose. Eval suites can assert on kind without flaky string matching. Logs become structured. New failure modes get new kinds rather than edits to prose all over the codebase.
</CheckQuestion>

<CheckQuestion id="m10-q5" question="5. When does semantic tool search help?">
When the catalog exceeds roughly 30 to 50 tools and the model starts missing relevant ones because each tool's description is competing for attention. Pre-filter to top-K by relevance (substring scoring or embedding similarity); show the model only the candidates. Below that threshold, full catalog plus good descriptions is enough.
</CheckQuestion>

<CheckQuestion id="m10-q6" question="6. A v2 of a tool ships. What happens to v1?">
Mark v1 deprecated in its description ("DEPRECATED: use v2 instead. Will be removed on YYYY-MM-DD"). Keep it working through a grace period. Communicate the migration via README and changelog. Do not silently change v1's behavior; that breaks downstream code that assumed the old shape.
</CheckQuestion>

<CheckQuestion id="m10-q7" question="7. Two tools, same job, different names. What goes wrong?">
The model picks inconsistently across runs because both descriptions fit. Eval suites become flaky; observability becomes hard to read; collaborators see different agent behavior than you do. Pick one, deprecate the other, and update prompts to use the canonical name.
</CheckQuestion>

<CheckQuestion id="m10-q8" question="8. Plan-then-execute: when to use it?">
When tool selection is ambiguous, the task is multi-step, or the agent's track record on similar tasks is mixed. The plan acts as a cognitive retrieval step that narrows the tool space before any actual tool_use block fires. Cost: 50 to 200 extra tokens per turn. Worth it for hard tasks; overhead for easy ones.
</CheckQuestion>
